## Abstract

This RFC defines the implementation strategy for optimizing Soothe's tool interfaces
to achieve performance comparable to Claude Code CLI. The core problems identified
are: (1) unified dispatch tools create cognitive load through mode/action indirection,
(2) lack of surgical code editing forces error-prone full-file rewrites, (3) stateless
Python execution prevents iterative data analysis, and (4) verbose tool descriptions
overwhelm the LLM.

This RFC introduces single-purpose tools with direct naming, surgical editing
capabilities, persistent Python sessions, and concise descriptions. The implementation
maintains backward compatibility through wrapper layers while providing a clear
migration path.

## Executive Summary

### Problem Statement

Users report that Soothe's LLM cannot effectively use tools. Analysis reveals:

- **Tool Call Success Rate**: ~60% (target: 95%)
- **Average Tool Calls per Task**: High (target: reduce 30%)
- **Common Failure Modes**:
  - Wrong mode parameter in `execute` tool
  - Confused action parameter in `workspace` tool
  - Missing parameters due to complex interfaces
  - Lost work from full-file rewrite errors

### Solution Overview

Replace unified dispatch tools with single-purpose tools:

```
Before (unified):          After (single-purpose):
execute(mode="shell")  →   run_command()
execute(mode="python") →   run_python()
workspace(action="read") → read_file()
workspace(action="write") → write_file()
```

Add surgical editing capabilities:

```
Before: Read full file → modify in memory → rewrite entire file
After:  edit_file_lines(start=45, end=50, new_content="...")
```

Add Python session persistence:

```
Before: Each call = fresh session, no variable persistence
After:  Variables persist across calls within same thread
```

### Expected Impact

- **Tool Call Success Rate**: 60% → 95%
- **Average Tool Calls per Task**: Reduce by 30%
- **Time to Task Completion**: Reduce by 50%
- **LLM Token Usage**: Reduce by 25% (simpler descriptions)

## Design Principles

### 1. Single Responsibility

Each tool does ONE thing and does it well.

**Bad**: `execute(code, mode, ...)` - multiple responsibilities
**Good**: `run_command(command)` - single responsibility

### 2. Direct Naming

Tool name immediately indicates purpose.

**Bad**: `execute(mode="shell")` - requires remembering mode
**Good**: `run_command()` - obvious from name

### 3. Minimal Parameters

Only required parameters, sensible defaults for rest.

**Bad**: `workspace(action="read", path, start_line, end_line, encoding, ...)`
**Good**: `read_file(path, start_line=None, end_line=None)`

### 4. Concise Descriptions

5-8 lines maximum, focusing on WHEN to use, not HOW.

### 5. Surgical Operations

Enable line-based editing instead of full-file rewrites.

### 6. Session Continuity

Python execution maintains state across calls.

### Backward Compatibility Wrapper Pattern

Old unified tools become thin wrappers:

```python
class ExecuteTool(BaseTool):
    """Deprecated: Use run_command, run_python, or run_background instead."""

    name = "execute"
    description = "DEPRECATED - Use specific tools: run_command, run_python, run_background"

    def _run(self, code: str, mode: str = "shell", **kwargs) -> str:
        import warnings
        warnings.warn(
            "execute tool is deprecated. Use run_command, run_python, or run_background.",
            DeprecationWarning,
            stacklevel=2
        )

        if mode == "shell":
            return RunCommandTool()._run(command=code, **kwargs)
        elif mode == "python":
            return RunPythonTool()._run(code=code, **kwargs)
        elif mode == "background":
            return RunBackgroundTool()._run(command=code, **kwargs)
        else:
            raise ValueError(f"Unknown mode: {mode}")
```

### Session Management for Python Execution

**Architecture**:

```
┌─────────────────────────────────────────────────────────────┐
│ Thread ID (from LangGraph config)                           │
├─────────────────────────────────────────────────────────────┤
│ Session Manager (singleton)                                 │
│  ├─ sessions: Dict[thread_id, InteractiveShell]             │
│  ├─ get_or_create(thread_id) → InteractiveShell             │
│  ├─ cleanup(thread_id)                                      │
│  └─ cleanup_all()                                           │
└─────────────────────────────────────────────────────────────┘
```

**Session Lifecycle**:

1. **Creation**: First `run_python` call in thread creates session
2. **Reuse**: Subsequent calls reuse same InteractiveShell instance
3. **Cleanup**: Thread termination triggers cleanup via SootheRunner

**Implementation**:

```python
from IPython.core.interactiveshell import InteractiveShell
from threading import Lock

class PythonSessionManager:
    _instance = None
    _lock = Lock()

    def __new__(cls):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._sessions = {}
        return cls._instance

    def get_or_create(self, session_id: str) -> InteractiveShell:
        if session_id not in self._sessions:
            self._sessions[session_id] = InteractiveShell.instance()
        return self._sessions[session_id]

    def cleanup(self, session_id: str):
        if session_id in self._sessions:
            shell = self._sessions.pop(session_id)
            shell.reset()

    def cleanup_all(self):
        for session_id in list(self._sessions.keys()):
            self.cleanup(session_id)
```

## Error Recovery Format

Standardized error response structure:

```python
{
    "error": str,  # Short error description
    "details": dict[str, Any],  # Context-specific details
    "suggestions": list[str],  # Recovery suggestions
    "recoverable": bool,  # Whether retry is possible
    "auto_retry_hint": Optional[str]  # Example command for retry
}
```

**Example**:

```python
{
    "error": "File already exists",
    "details": {
        "file": "/path/to/file.py",
        "action": "write"
    },
    "suggestions": [
        "Use read_file first to check current contents",
        "Use edit_file_lines to modify specific sections",
        "Use write_file with mode='overwrite' to replace entirely"
    ],
    "recoverable": True,
    "auto_retry_hint": "read_file(path='/path/to/file.py')"
}
```

## Appendix: Workflow Comparison

### Modifying a Function (Before vs After)

**Before (5 steps, error-prone)**:

```
1. workspace(action="read", path="utils.py")
   → Receive 500-line file

2. LLM must parse entire file in memory
   → High token cost, risk of truncation

3. LLM constructs modified full file
   → Risk of losing context, introducing errors

4. workspace(action="write", path="utils.py", content=FULL_FILE, overwrite=True)
   → Any error corrupts entire file

5. workspace(action="read", path="utils.py")
   → Verify change
```

**After (2 steps, safe)**:

```
1. edit_file_lines(path="utils.py", start_line=45, end_line=50, new_content="...")
   → Only touch lines 45-50

2. read_file(path="utils.py", start_line=40, end_line=55)
   → Verify change (only 15 lines to check)
```

### Iterative Data Analysis (Before vs After)

**Before (broken)**:

```
1. execute(code="import pandas as pd\ndf = pd.read_csv('data.csv')", mode="python")
   → Success

2. execute(code="df.head()", mode="python")
   → ERROR: df not defined (fresh session)
```

**After (works naturally)**:

```
1. run_python(code="import pandas as pd\ndf = pd.read_csv('data.csv')")
   → Success, df in session

2. run_python(code="df.head()")
   → Works! df persists from previous call

3. run_python(code="df.groupby('category').sum()")
   → Continue analysis seamlessly
```
