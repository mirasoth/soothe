# RFC-0014: Capability Abstraction and Tool Consolidation

**RFC**: 0014
**Title**: Capability Abstraction and Tool Consolidation
**Status**: Draft
**Kind**: Architecture Design
**Created**: 2026-03-19
**Dependencies**: RFC-0001, RFC-0002, RFC-0008

## Abstract

This RFC defines an architecture for consolidating Soothe's tool and subagent
surface into a minimal set of **capability-level** tools. The core insight is
the **Capability Abstraction Principle**: when multiple tools serve the same
capability domain, expose a single unified tool to the LLM and push
implementation routing into deterministic system logic. This reduces the LLM's
decision space from ~22 individual tools to 6 capability tools, improving
accuracy, reducing token cost, and eliminating overlapping tool descriptions.

The design also introduces **domain-scoped prompt guidance**, extending
`UnifiedClassification` (RFC-0008) with a `capability_domains` field so the
system injects only the relevant tool guidance for each query.

## Motivation

### Problem: Tool Proliferation

Soothe currently exposes individual tool classes directly to the LLM:

| Group        | Individual tools exposed          | Count |
|--------------|-----------------------------------|-------|
| file_edit    | create_file, read_file, delete_file, list_files, search_in_files, get_file_info | 6 |
| cli          | run_cli, get_current_directory, list_directory, run_cli_background, kill_process, check_command_exists | 6 |
| python_exec  | execute_python_code               | 1 |
| tabular      | get_tabular_columns, get_data_summary, validate_data_quality | 3 |
| document     | document_qa, extract_text, get_document_info | 3 |
| wizsearch    | wizsearch_search, wizsearch_crawl_page | 2 |
| inquiry      | inquiry                           | 1 |
| datetime     | current_datetime                  | 1 |
| **Total**    |                                   | **~23** |

Research shows LLM tool-selection accuracy degrades significantly beyond
15 tools. The current set contains substantial overlap (e.g., `list_files` vs
`list_directory`, `search_in_files` vs `run_cli("grep ...")`) and forces the
LLM to make implementation-level choices it is poorly suited for.

### Problem: Monolithic Prompt Guidance

The current `_TOOL_ORCHESTRATION_GUIDE` injects all tool guidance for every
query, wasting tokens on chitchat and simple questions that need no tool
awareness.

### Solution: Capability Abstraction Principle

> When the agent has N tools that serve the same capability domain, expose a
> single unified tool at the agent level and push routing logic into the tool
> implementation. The agent decides **what capability** to invoke; the system
> decides **how** to fulfil it.

```
┌──────────────────────────────────┬──────────────────────────────────────────────────┐
│ What the LLM decides             │ What the system decides                          │
├──────────────────────────────────┼──────────────────────────────────────────────────┤
│ "I need to search the web"       │ Which search engine to use                       │
│ "I need to read code"            │ Whether to use file_edit, grep, or cli           │
│ "I need to extract page content" │ Whether to use jina, wizsearch crawl, or browser │
│ "I need to inspect data"         │ Whether to use tabular or document tools         │
└──────────────────────────────────┴──────────────────────────────────────────────────┘
```

## Design

### 1. Capability Domain Taxonomy

Six domains map the full tool landscape:

| Domain       | Exposed Tool  | Internal Implementations              |
|--------------|---------------|---------------------------------------|
| **Research** | `research`    | InquiryEngine → WebSource, AcademicSource, FilesystemSource, CLISource, BrowserSource, DocumentSource |
| **Websearch**| `websearch`   | WizsearchSearchTool (single-call, no iteration) |
| **Workspace**| `workspace`   | CreateFileTool, ReadFileTool, DeleteFileTool, ListFilesTool, SearchInFilesTool, GetFileInfoTool, ListDirTool |
| **Execute**  | `execute`     | CliTool, PythonExecutorTool, RunCliBackgroundTool, KillProcessTool, GetCurrentDirTool, CheckCommandExistsTool |
| **Data**     | `data`        | TabularColumnsTool, TabularSummaryTool, TabularQualityTool, DocumentQATool, ExtractTextTool, GetDocumentInfoTool |
| **Datetime** | `datetime`    | CurrentDateTimeTool (unchanged) |

### 2. Consolidated Tool Interfaces

#### 2.1 `workspace` Tool

Consolidates file_edit (6 tools) and cli's list_directory into a single
action-dispatched tool.

```
workspace(action, path, ...)
  action: "read" | "write" | "delete" | "search" | "list" | "info"
```

Internal dispatch:
- `read` → ReadFileTool
- `write` → CreateFileTool
- `delete` → DeleteFileTool
- `search` → SearchInFilesTool
- `list` → ListFilesTool (with fallback to ListDirTool)
- `info` → GetFileInfoTool

#### 2.2 `execute` Tool

Consolidates cli (5 remaining tools) and python_executor into a mode-dispatched
tool.

```
execute(code, mode, ...)
  mode: "shell" | "python" | "background"
```

Internal dispatch:
- `shell` → CliTool (also handles `get_current_directory`, `check_command_exists` via command routing)
- `python` → PythonExecutorTool
- `background` → RunCliBackgroundTool

Special commands (detected by prefix):
- `kill <pid>` → KillProcessTool

#### 2.3 `data` Tool

Consolidates tabular and document tools, routing by file extension.

```
data(file_path, question, operation)
  operation: "inspect" | "summary" | "quality" | "extract" | "info"
```

Internal dispatch based on file extension:
- `.csv`, `.tsv`, `.xlsx`, `.xls`, `.json`, `.parquet` → tabular tools
- `.pdf`, `.docx`, `.txt`, `.md`, `.rst` → document tools

#### 2.4 `websearch` Tool (renamed from wizsearch)

Exposes only the search capability. Page crawling moves to `research` tool's
BrowserSource.

```
websearch(query, max_results_per_engine, timeout_seconds)
```

The tool is renamed from `wizsearch_search` to `websearch` for clarity.
Uses configured search engines from config.yml (no engines parameter exposed).

#### 2.5 `research` Tool (renamed from inquiry)

The InquiryEngine tool, renamed for user clarity.

```
research(topic, domain)
  domain: "web" | "code" | "deep" | "auto"
```

Replaces:
- ~~`research` subagent~~ → `research(domain='web')` (removed)
- ~~`scout` subagent~~ → `research(domain='code')` (removed)

### 3. Subagent Surface Reduction

| Subagent   | Status                  | Rationale                         |
|------------|-------------------------|-----------------------------------|
| browser    | Keep                    | Unique: interactive JS, login, forms |
| claude     | Keep                    | Unique: superior reasoning        |
| skillify   | Keep                    | Unique: skill discovery           |
| weaver     | Keep                    | Unique: agent generation          |

The research and scout subagents have been removed from the codebase entirely.
Their functionality is now provided by the `research` tool with domain parameters.

### 4. Domain-Scoped Prompt Guidance

#### 4.1 Extended Classification

`UnifiedClassification` gains a `capability_domains` field:

```python
class UnifiedClassification(BaseModel):
    task_complexity: Literal["chitchat", "medium", "complex"]
    is_plan_only: bool
    template_intent: ...
    capability_domains: list[Literal[
        "research", "workspace", "execute", "data", "browse", "reason", "compose"
    ]]
```

#### 4.2 Layered Injection

`SystemPromptOptimizationMiddleware` injects guidance per domain:

- **chitchat** → zero tool guidance (saves ~500 tokens)
- **"search for latest AI news"** → `_RESEARCH_GUIDE` only
- **"refactor the auth module"** → `_WORKSPACE_GUIDE` + `_EXECUTE_GUIDE`
- **"research then implement"** → all relevant guides

Each domain guide is 3-5 lines (vs current monolithic 40-line block).

### 5. Tool Count Impact

| Before  | After       |
|---------|-------------|
| 23 tools + 6 subagents | 6 tools + 4 subagents |

Well below the 15-tool accuracy degradation threshold.

### 6. Default Configuration

```yaml
tools:
  - datetime       # Current date/time
  - workspace      # File operations (read, write, search, list, delete, info)
  - execute        # Run shell commands, Python code, background processes
  - data           # Inspect CSV, Excel, PDF, DOCX, etc.
  - websearch      # Quick single-call web search
  - research       # Deep multi-source iterative research
```

Standalone `wikipedia`, `arxiv`, `serper`, `jina` are removed from defaults
(absorbed into `research` tool's InquiryEngine sources). They remain available
as explicit opt-in tool groups for backward compatibility.

### 7. Backward Compatibility

- Old tool group names (`file_edit`, `cli`, `python_executor`, `tabular`,
  `document`, `wizsearch`, `inquiry`) continue to work in config and resolve
  to their original individual tools for users who prefer fine-grained control.
- The `research` and `scout` subagent factories remain registered; only prompt
  guidance changes.

## Open Questions

1. Should `workspace` support a `rename` action?
2. Should `execute(mode='shell')` auto-detect `pwd` and `which` commands to
   avoid needing separate GetCurrentDirTool and CheckCommandExistsTool?
