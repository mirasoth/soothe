# RFC-0010: Failure Recovery, Progressive Persistence, and Artifact Storage

**RFC**: 0010
**Title**: Failure Recovery, Progressive Persistence, and Artifact Storage
**Status**: Draft
**Created**: 2026-03-18
**Updated**: 2026-03-18
**Related**: RFC-0001, RFC-0002, RFC-0007, RFC-0009

## Abstract

This RFC introduces a failure recovery mechanism for the Soothe runner that
enables resumption of interrupted step loops and goal DAGs after a crash.
It adds progressive persistence (checkpointing after each step and goal
completion), a structured artifact store for human-reviewable run outputs,
goal DAG state restoration on resume, and a cross-validated synthesized
report for the root goal.

Key design decisions:

- **Hard-cut storage consolidation**: all per-run data lives under
  `$SOOTHE_HOME/runs/{thread_id}/`.  The legacy `threads/` directory is
  removed with no backward compatibility code.
- **DurabilityProtocol slimming**: `save_state`/`load_state` are removed
  from `DurabilityProtocol`.  The runner uses `RunArtifactStore` directly
  for checkpoints and artifacts.
- **Hierarchical artifact layout**: reports and produced files are
  organized by `goals/{goal_id}/steps/{step_id}/`.
- **Reference + selective copy**: workspace files (code edits) are tracked
  by path reference; reports and downloaded artifacts are copied into the
  run directory.

## Motivation

### Problem 1: All In-Flight State Is Lost on Crash

RFC-0009 introduced DAG-based step and goal execution.  However, the
runner only calls `save_state` once -- at the very end of execution.
If the process crashes mid-plan or mid-goal-DAG, all progress is lost:

| Data | Persisted? | Restored on Resume? |
|------|-----------|---------------------|
| Thread metadata | Yes | Yes (via `resume_thread`) |
| Durability state (goals, last_query) | Yes (at end only) | **No** (`load_state` never called) |
| GoalEngine (goal DAG, status, depends_on) | Yes (at end only) | **No** (`restore_from_snapshot` never called) |
| Plan / PlanStep results | **No** | No |
| StepScheduler progress | **No** | No |
| Context ledger | Yes | Yes (via `context.restore`) |
| LangGraph checkpoints | Yes (PostgreSQL) | Yes (via thread_id) |

The `GoalEngine.restore_from_snapshot()` API exists but is never invoked.

### Problem 2: No Structured Artifact Storage

Intermediate goal results are richer than text strings:

- **Text reports** -- analysis, summaries (currently truncated to 2000 chars)
- **Downloaded files** -- PDFs, images, datasets (written to CWD, untracked)
- **Code patches** -- generated by subagents (written to workspace, untracked)
- **Structured data** -- JSON, CSV produced by tools (lost or scattered)

There is no per-run directory, no manifest, and no way for humans to
browse or reuse outputs from completed goals.

### Problem 3: No Cross-Validation in Final Report

When a goal DAG completes, the `GoalReport.summary` is simply
`response_text[:500]`.  There is no synthesis, cross-validation between
parallel step results, or consistency checking between dependent goals.

### Design Goals

1. **Progressive persistence**: checkpoint after each step and goal.
2. **Goal DAG recovery**: restore goal engine state on thread resume.
3. **Plan recovery**: restore plan and skip completed steps on resume.
4. **Structured artifacts**: human-browsable run outputs in `$SOOTHE_HOME/runs/`.
5. **Daemon restart recovery**: detect and log incomplete threads on start.
6. **Cross-validated final report**: LLM-synthesized summary.

## Architecture

### Storage Layout

```
$SOOTHE_HOME/
  runs/{thread_id}/               # Per-run directory (NEW)
    conversation.jsonl            # Conversation log (was threads/{tid}.jsonl)
    checkpoint.json               # CheckpointEnvelope (recovery state)
    manifest.json                 # Artifact index
    goals/
      {goal_id}/
        report.json               # GoalReport (machine-readable)
        report.md                 # Human-readable goal summary
        steps/
          {step_id}/
            report.json           # StepReport (machine-readable)
            report.md             # Human-readable step output
            artifacts/            # Produced files (copied here)
  durability/data/                # KEEP -- thread metadata + index only
    thread_{thread_id}.json
    thread_index.json
  logs/soothe.log                 # KEEP -- global log
```

### DurabilityProtocol Changes

`save_state` and `load_state` are removed from `DurabilityProtocol`.
`DurabilityProtocol` retains its core thread lifecycle responsibility:
`create_thread`, `resume_thread`, `suspend_thread`, `archive_thread`,
`list_threads`.

The runner uses `RunArtifactStore` directly for checkpoints and artifacts.
This provides clean separation of concerns:

- `DurabilityProtocol` = thread lifecycle management
- `RunArtifactStore` = run artifacts, checkpoints, reports, manifest

### RunArtifactStore

New component at `core/artifact_store.py`:

```python
class RunArtifactStore:
    """Manages $SOOTHE_HOME/runs/{thread_id}/ directory."""

    def __init__(self, thread_id: str, soothe_home: str = SOOTHE_HOME) -> None: ...

    @property
    def run_dir(self) -> Path: ...

    @property
    def conversation_log_path(self) -> Path: ...

    def ensure_step_dir(self, goal_id: str, step_id: str) -> Path: ...
    def write_step_report(self, goal_id: str, step: PlanStep, duration_ms: int) -> None: ...
    def write_goal_report(self, report: GoalReport) -> None: ...
    def record_artifact(self, entry: ArtifactEntry) -> None: ...
    def save_checkpoint(self, envelope: dict[str, Any]) -> None: ...
    def load_checkpoint(self) -> dict[str, Any] | None: ...
    def save_manifest(self) -> None: ...
    def load_manifest(self) -> RunManifest | None: ...
```

`save_checkpoint()` writes atomically (write to tmp, then rename).
`write_step_report()` and `write_goal_report()` produce both `.json` and
`.md` files.

### Artifact Tracking

Two categories:

- **Produced artifacts**: Reports (StepReport, GoalReport) are always
  written as `.json` + `.md`.  Downloaded files and explicitly generated
  outputs are copied into `steps/{step_id}/artifacts/`.
- **Referenced artifacts**: Workspace files created/modified by tools
  (`write_file`, `edit_file`) are tracked in `manifest.json` by original
  workspace path -- NOT copied (avoids duplicating code repos).

```python
class ArtifactEntry(BaseModel):
    path: str                           # Relative to run dir
    source: Literal["produced", "reference"]
    original_path: str = ""             # Workspace path for references
    tool_name: str = ""
    step_id: str = ""
    goal_id: str = ""
    size_bytes: int = 0

class RunManifest(BaseModel):
    version: int = 1
    thread_id: str
    created_at: str
    updated_at: str
    query: str = ""
    mode: Literal["single_pass", "autonomous"] = "single_pass"
    status: Literal["in_progress", "completed", "failed"] = "in_progress"
    goals: list[str] = Field(default_factory=list)
    artifacts: list[ArtifactEntry] = Field(default_factory=list)
```

### Persistence Points

```
Non-Autonomous:
  _pre_stream
    |
    step_loop:
      for each batch:
        for each step:
          _execute_step()
          artifact_store.write_step_report()
          artifact_store.save_checkpoint()
        |
    _post_stream
      artifact_store.save_checkpoint(status="completed")

Autonomous:
  _pre_stream
    |
    goal_loop:
      for each goal batch:
        for each goal:
          plan_creation
          step_loop (per-step checkpoints as above)
          reflection
          artifact_store.write_goal_report()
          artifact_store.save_checkpoint()
        |
    artifact_store.save_checkpoint(status="completed")
```

### CheckpointEnvelope

```python
class CheckpointEnvelope(BaseModel):
    version: int = 1
    timestamp: str = ""
    mode: Literal["single_pass", "autonomous"] = "single_pass"
    last_query: str = ""
    thread_id: str = ""
    goals: list[dict[str, Any]] = Field(default_factory=list)
    active_goal_id: str | None = None
    plan: dict[str, Any] | None = None
    completed_step_ids: list[str] = Field(default_factory=list)
    total_iterations: int = 0
    status: Literal["in_progress", "completed", "failed"] = "in_progress"
```

### Recovery Flow

```
Thread Resume (_pre_stream):
  1. resume_thread(thread_id)           ← existing
  2. context.restore(thread_id)          ← existing
  3. envelope = artifact_store.load_checkpoint()   ← NEW
  4. if envelope and envelope.status == "in_progress":
       goal_engine.restore_from_snapshot(envelope.goals)
       plan = Plan.model_validate(envelope.plan)
       mark completed steps from envelope.completed_step_ids
       emit soothe.recovery.resumed
  5. StepScheduler.ready_steps() skips completed steps
```

### Recovery Scenarios

**Scenario 1: Crash mid-step-loop (non-autonomous)**

```
Before crash:
  Step 1: completed (checkpoint saved, report written)
  Step 2: completed (checkpoint saved, report written)
  Step 3: in_progress (not saved)
  Step 4: pending

After resume:
  Step 1: completed (restored from checkpoint, report on disk)
  Step 2: completed (restored from checkpoint, report on disk)
  Step 3: pending (re-executed)
  Step 4: pending
```

**Scenario 2: Crash mid-goal-DAG (autonomous)**

```
Before crash:
  Goal A: completed (checkpoint saved, GoalReport on disk)
  Goal B: in_progress, Step 1 completed, Step 2 in_progress
  Goal C: pending (depends on A, B)

After resume:
  Goal A: completed (restored, report on disk for human review)
  Goal B: active, Step 1 completed (restored), Step 2+ pending
  Goal C: pending (deps not yet met)
```

**Scenario 3: Daemon restart**

Scan `$SOOTHE_HOME/runs/*/checkpoint.json` for `status="in_progress"`.
Log incomplete threads so the user can resume them.

### Cross-Validated Final Report

When a goal completes, the runner generates a synthesized summary:

```python
async def _synthesize_root_goal_report(
    self,
    goal: Goal,
    step_reports: list[StepReport],
    child_goal_reports: list[GoalReport],
) -> str:
```

The synthesis prompt asks the LLM to:
1. Summarize what was accomplished across all steps/goals
2. Cross-validate: note contradictions or conflicting information
3. Identify gaps: what information is missing or incomplete
4. State confidence level based on source agreement

Heuristic fallback when LLM is unavailable.

### Enhanced Reflection

The `Reflection` model gains dependency-aware fields:

```python
class Reflection(BaseModel):
    assessment: str
    should_revise: bool
    feedback: str
    blocked_steps: list[str] = Field(default_factory=list)
    failed_details: dict[str, str] = Field(default_factory=dict)
```

`reflect()` implementations distinguish "directly failed" from
"blocked by dependency" steps.

## Constraints

- Recovery does NOT replay LangGraph steps; it skips completed Soothe steps.
- LangGraph's own checkpointing handles conversation state per thread_id.
- The two systems are complementary: Soothe checkpoints track plan/goal
  progress; LangGraph checkpoints track agent conversation state.
- Cross-validated synthesis requires a planner LLM; heuristic fallback is
  always available.
