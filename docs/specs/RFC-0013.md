# RFC-0013: Unified Daemon Communication Protocol

**RFC**: 0013
**Title**: Unified Daemon Communication Protocol for Multi-Transport IPC
**Status**: Draft
**Kind**: Architecture Design
**Created**: 2026-03-19
**Dependencies**: RFC-0001, RFC-0002, RFC-0003

## Abstract

This RFC defines a transport-agnostic daemon communication protocol that supports Unix domain sockets, WebSockets, and HTTP REST. The protocol specifies a common JSON-based message format, security requirements, and implementation interface. This architecture enables the Soothe daemon to serve both local CLI/TUI clients (via Unix socket) and remote/web clients (via WebSocket/HTTP) using the same protocol layer, while maintaining backward compatibility with existing clients.

## Motivation

### Problem: Single Transport Limitation

RFC-0003 defines the daemon IPC architecture using Unix domain sockets exclusively. This design has limitations:

1. **Browser incompatibility**: Web browsers cannot connect to Unix sockets, preventing web-based UI development
2. **Remote access**: Unix sockets are local-only, requiring SSH tunneling or local presence for remote access
3. **Mobile clients**: Mobile applications cannot connect to Unix sockets
4. **Infrastructure constraints**: Cloud deployments and containerized environments may prefer HTTP-based protocols

### Problem: No Protocol Specification

The current implementation in `src/soothe/cli/daemon/protocol.py` provides encoding/decoding utilities but lacks:
- Formal message schema definitions
- Transport abstraction layer
- Security requirements for different transport modes
- Versioning and compatibility guidelines

### Design Goals

1. **Transport independence**: Same message protocol over Unix socket, WebSocket, and HTTP REST
2. **Backward compatibility**: Existing Unix socket clients continue working unchanged
3. **Simplicity by default**: No built-in authentication - handled by external services
4. **Protocol evolution**: Versioning and extension mechanisms
5. **Clear specification**: Formal message schemas and implementation requirements

### Non-Goals

- **Authentication and authorization**: Handled by external services (reverse proxies, API gateways, etc.)
- **User management**: Not within Soothe's scope
- **Multi-tenancy**: Soothe is single-tenant by design

## Guiding Principles

### Principle 1: Protocol-Transport Separation

The message protocol (what we send) is independent of the transport mechanism (how we send it). Unix socket, WebSocket, and HTTP REST all use the same JSON message format, with transport-specific framing.

### Principle 2: Minimal Wire Overhead

The protocol uses JSON for simplicity and debuggability, not binary formats. Message framing is minimal:
- Unix socket: newline-delimited JSON lines
- WebSocket: native text frames (no newline needed)
- HTTP REST: standard HTTP request/response

### Principle 3: Streaming-First Design

The protocol is designed for bidirectional streaming (Unix socket, WebSocket), not request-response. HTTP REST provides stateless CRUD operations. Clients and servers exchange asynchronous messages over persistent connections.

### Component Responsibilities

#### Unix Socket Server

**Responsibility**: Serve local CLI/TUI clients over Unix domain socket.

**Capabilities**:
- Path: `$SOOTHE_HOME/soothe.sock` (default: `~/.soothe/soothe.sock`)
- Security: OS-level file permissions (user-only by default)
- Performance: ~0.1ms latency, ~1GB/s throughput
- Backward compatibility: Maintains existing behavior from RFC-0003

#### WebSocket Server

**Responsibility**: Serve web-based and remote clients over WebSocket protocol.

**Capabilities**:
- Default host: `127.0.0.1` (localhost only by default)
- Default port: `8765`
- Security: CORS validation, connection limits
- Performance: ~0.5ms latency (localhost), ~10-50ms (remote)

**Security Model**: For production deployments, use a reverse proxy to handle authentication, TLS, and rate limiting.

#### Protocol Handler

**Responsibility**: Transport-agnostic message processing.

**Capabilities**:
- Message validation (schema checking)
- Message routing to appropriate handlers
- Broadcast to all connected clients
- Client lifecycle management

#### Message Router

**Responsibility**: Route messages to SootheRunner and broadcast events.

**Capabilities**:
- Route client inputs to runner
- Stream runner events to clients
- Handle slash commands
- Thread management operations

### Event Bus Architecture

#### Overview

The daemon uses a pub/sub event bus architecture to route events to connected clients based on thread subscriptions. This provides client isolation and prevents event mixing when multiple clients are connected simultaneously.

#### Component Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                        Event Bus Core                             │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  EventBus                                                    │  │
│  │  - publish(topic: str, event: dict)                         │  │
│  │  - subscribe(topic: str, queue: asyncio.Queue)              │  │
│  │  - unsubscribe(topic: str, queue: asyncio.Queue)            │  │
│  └────────────────────────────────────────────────────────────┘  │
│       ↓                                                           │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  ClientSessionManager                                        │  │
│  │  - sessions: dict[client_id, ClientSession]                 │  │
│  │  - create_session(transport, client_info) → client_id       │  │
│  │  - subscribe_thread(client_id, thread_id)                   │  │
│  │  - remove_session(client_id)                                │  │
│  └────────────────────────────────────────────────────────────┘  │
│       ↓                                                           │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  ClientSession                                               │  │
│  │  - client_id: str                                            │  │
│  │  - transport: TransportServer                                │  │
│  │  - transport_client: Any  # Transport-specific client obj   │  │
│  │  - subscriptions: set[str]  # thread_ids                     │  │
│  │  - event_queue: asyncio.Queue[dict] (maxsize=100)          │  │
│  │  - sender_task: asyncio.Task  # Sends events to client      │  │
│  └────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

#### Event Flow

**Query Execution Flow**:
```
SootheRunner.astream()
  → emit event with thread_id
  → EventBus.publish(f"thread:{thread_id}", event)
  → EventBus routes to subscribed client queues
  → ClientSession.sender_task sends to transport
  → Transport.send(client, event)
```

**Client Subscription Flow**:
```
Client connects
  → Transport creates session via ClientSessionManager.create_session()
  → Client sends subscribe_thread message
  → ClientSessionManager.subscribe_thread(client_id, thread_id)
  → EventBus.subscribe(f"thread:{thread_id}", client_queue)
  → Client only receives events for that thread
```

#### Topic-Based Routing

Events are published to topics in the format `thread:{thread_id}`:
- Each thread gets a dedicated topic
- Clients subscribe to topics they're interested in
- Multiple clients can subscribe to the same thread
- Unsubscribed events are discarded (no global topic)

#### Client Isolation

Each client has:
- Unique `client_id` (UUID)
- Dedicated event queue (maxsize=100, drops old events when full)
- Independent sender task
- Set of thread subscriptions

Events from different threads are isolated per client - no cross-contamination.

### Architectural Constraints

1. **Single daemon instance**: Only one daemon can run per user (PID lock enforcement)
2. **Unified message schema**: All transports use identical JSON message format
3. **Transport abstraction**: Protocol layer has no transport-specific logic
4. **No built-in authentication**: Security handled by external services
5. **Thread subscription required**: All clients MUST subscribe to threads to receive events (BREAKING CHANGE)
6. **Client isolation enforced**: Events are routed to subscribed clients only

### Abstract Schemas

#### Transport Configuration

**TransportMode**: Enumeration of supported transport types
- `UNIX_SOCKET` - Unix domain socket for local clients
- `WEBSOCKET` - WebSocket for remote/web clients
- `HTTP_REST` - HTTP REST API for CRUD operations

**UnixSocketConfig**: Unix domain socket configuration
- `enabled`: Enable/disable Unix socket server
- `path`: Socket file path (default: `~/.soothe/soothe.sock`)

**WebSocketConfig**: WebSocket server configuration
- `enabled`: Enable/disable WebSocket server
- `host`: Bind address (default: `127.0.0.1`)
- `port`: Listen port (default: `8765`)
- `tls_enabled`: Enable TLS encryption
- `tls_cert`, `tls_key`: TLS certificate paths

**HttpRestConfig**: HTTP REST server configuration
- `enabled`: Enable/disable HTTP REST server
- `host`: Bind address (default: `127.0.0.1`)

## Protocol Specification

### Message Format

All messages are JSON objects with a required `type` field.

#### Base Message Schema

```json
{
  "type": "string (required)",
  "...": "additional fields based on type"
}
```

### Client → Server Messages

#### Input Message

Send user input to the daemon for processing.

```json
{
  "type": "input",
  "text": "string (required)",
  "autonomous": "boolean (optional, default: false)",
  "max_iterations": "integer (optional, default: null)"
}
```

**Example**:
```json
{
  "type": "input",
  "text": "Analyze the authentication code",
  "autonomous": false
}
```

#### Command Message

Send a slash command to the daemon.

```json
{
  "type": "command",
  "cmd": "string (required)"
}
```

**Example**:
```json
{
  "type": "command",
  "cmd": "/help"
}
```

**Valid commands**:
- `/help`, `/exit`, `/quit`, `/detach`
- `/plan`, `/memory`, `/context`, `/policy`, `/history`, `/review`
- `/resume` (interactive thread selection)
- `/clear`, `/config`

#### Resume Thread Message

Request the daemon to resume a specific conversation thread.

```json
{
  "type": "resume_thread",
  "thread_id": "string (required)"
}
```

**Example**:
```json
{
  "type": "resume_thread",
  "thread_id": "abc123-def456-ghi789"
}
```

**Behavior**:
- Loads thread state from durability backend
- Sets thread as active for the client
- Does NOT automatically subscribe the client to the thread
- Client MUST send `subscribe_thread` message separately to receive events

#### Subscribe Thread Message

Subscribe the client to receive events for a specific thread.

```json
{
  "type": "subscribe_thread",
  "thread_id": "string (required)"
}
```

**Example**:
```json
{
  "type": "subscribe_thread",
  "thread_id": "abc123-def456-ghi789"
}
```

**Behavior**:
- Registers client subscription with EventBus
- Client will receive all events for the specified thread
- Multiple subscriptions allowed (client can subscribe to multiple threads)
- Returns `subscription_confirmed` message on success

**Required Sequence**:
1. Client connects
2. Client sends `resume_thread` or `new_thread`
3. Client sends `subscribe_thread` with the thread_id
4. Client receives `subscription_confirmed`
5. Client now receives events for that thread

#### Detach Message

Notify the daemon that the client is detaching (daemon keeps running).

```json
{
  "type": "detach"
}
```

### Server → Client Messages

#### Status Message

Daemon state change notification (sent on connect and state transitions).

```json
{
  "type": "status",
  "state": "string (required: idle|running|stopped|stopping|detached)",
  "thread_id": "string (required, may be empty)",
  "client_id": "string (required, assigned on connection)",
  "input_history": "array of strings (optional, last 100 entries)"
}
```

**Example**:
```json
{
  "type": "status",
  "state": "running",
  "thread_id": "abc123",
  "client_id": "client-uuid-12345",
  "input_history": ["query 1", "query 2", "query 3"]
}
```

**States**:
- `idle`: Daemon is ready and waiting for input
- `running`: Daemon is processing a query
- `stopped`: Daemon is shutting down
- `stopping`: Daemon received stop signal
- `detached`: Client has detached (daemon still running)

#### Subscription Confirmed Message

Confirmation that client subscription to thread was successful.

```json
{
  "type": "subscription_confirmed",
  "thread_id": "string (required)",
  "client_id": "string (required)"
}
```

**Example**:
```json
{
  "type": "subscription_confirmed",
  "thread_id": "abc123",
  "client_id": "client-uuid-12345"
}
```

**Behavior**:
- Sent in response to `subscribe_thread` message
- Client will now receive all events for the specified thread
- If thread doesn't exist, returns `error` message instead

#### Event Message

Stream event from SootheRunner (canonical `(namespace, mode, data)` format).

```json
{
  "type": "event",
  "thread_id": "string (required, the thread this event belongs to)",
  "namespace": "array of strings (required)",
  "mode": "string (required: messages|updates|custom)",
  "data": "any (required, based on mode)"
}
```

**Example (messages mode)**:
```json
{
  "type": "event",
  "thread_id": "abc123",
  "namespace": [],
  "mode": "messages",
  "data": ["AIMessage content...", {"langgraph_metadata": {}}]
}
```

**Example (custom mode)**:
```json
{
  "type": "event",
  "thread_id": "abc123",
  "namespace": [],
  "mode": "custom",
  "data": {
    "type": "soothe.plan.step_started",
    "step_id": "step_1",
    "description": "Analyze codebase structure"
  }
}
```

**Event Routing**:
- Events are published to topics: `thread:{thread_id}`
- Only clients subscribed to the thread receive the event
- Unsubscribed clients do NOT receive any events
- Events include `thread_id` at the top level for routing

**Event Types**: See RFC-0003 "Stream Events" section for complete event taxonomy.

#### Command Response Message

Output from slash command execution.

```json
{
  "type": "command_response",
  "content": "string (required)"
}
```

**Example**:
```json
{
  "type": "command_response",
  "content": "Available commands:\n/help - Show commands\n/exit - Exit TUI\n..."
}
```

#### Error Message

Protocol error notification.

```json
{
  "type": "error",
  "code": "string (required)",
  "message": "string (required)",
  "details": "object (optional)"
}
```

**Example**:
```json
{
  "type": "error",
  "code": "INVALID_MESSAGE",
  "message": "Message missing required field: type",
  "details": {"received": {"text": "hello"}}
}
```

**Error Codes**:
- `INVALID_MESSAGE`: Malformed JSON or missing required fields
- `RATE_LIMITED`: Too many messages (if rate limiting is configured)
- `INTERNAL_ERROR`: Server-side error

## Transport Layer Specification

### Unix Domain Socket

**Wire Format**: Newline-delimited JSON lines (JSONL)

**Message Framing**:
```
<json-message>\n
```

**Connection Lifecycle**:
1. Client connects to `$SOOTHE_HOME/soothe.sock`
2. Server immediately sends initial `status` message
3. Client sends messages, each terminated with `\n`
4. Server sends messages, each terminated with `\n`
5. Connection closes on EOF or error

**Current Status**: ✅ **Fully Implemented**
- Reference: `src/soothe/cli/daemon/server.py`, `client.py`, `protocol.py`

### WebSocket

**Wire Format**: WebSocket text frames (RFC 6455)

**Message Framing**:
```
WebSocket Frame:
  - opcode: 0x01 (text)
  - payload: <json-message>  (no newline needed)
```

**Connection Lifecycle**:
1. Client initiates WebSocket handshake at `ws://host:port` or `wss://host:port`
2. Server accepts handshake, optionally validates CORS origin
3. Server sends initial `status` message
4. Client and server exchange JSON messages in text frames
5. Connection closes with WebSocket close frame (code 1000 for normal)

**Current Status**: ❌ **Not Implemented**
- Planned: Phase 2 of multi-transport architecture
- Estimated effort: 3-5 days

### HTTP REST Server

**Wire Format**: HTTP/1.1 with JSON request/response bodies

**Message Framing**: Standard HTTP request/response cycle with Content-Length headers

**Default Port**: `8766` (configurable, localhost only by default)

**Connection Lifecycle**:
1. Client sends HTTP request to `http://host:port/api/v1/endpoint`
2. Server validates CORS origin (if configured)
3. Server processes request and returns JSON response
4. Connection closes after response (stateless)

**Base URL**: `http://localhost:8766/api/v1`

**Transport Configuration Schema**:
- `enabled`: Enable/disable HTTP REST server
- `host`: Bind address (default: `127.0.0.1`)
- `port`: Listen port (default: `8766`)
- `tls_enabled`: Enable TLS encryption
- `tls_cert`, `tls_key`: TLS certificate paths
- `cors_origins`: Allowed CORS origins

**Current Status**: ❌ **Not Implemented**
- Planned: Phase 1 of multi-transport architecture
- Framework: FastAPI with Uvicorn ASGI server
- Estimated effort: 5-7 days

## REST API Specification

### Overview

The HTTP REST API provides CRUD operations for thread management, configuration, file operations, and system monitoring. It complements the WebSocket protocol for real-time streaming with standard HTTP patterns suitable for web applications and mobile clients.

**Design Principles**:
- RESTful resource naming conventions
- JSON request/response bodies
- Async FastAPI implementation for performance
- OpenAPI documentation auto-generated
- Shared authentication with WebSocket
- CORS support for web clients

## WebSocket + REST Integration

### When to Use Which Protocol

**Use WebSocket when**:
- Real-time streaming is required (active agent execution)
- Bidirectional communication needed
- Low-latency event updates required
- Client is actively monitoring agent progress

**Use HTTP REST when**:
- Fetching thread lists, configuration, or historical data
- Uploading or downloading files
- Performing CRUD operations (create, update, delete)
- Health checks and monitoring
- One-time queries without streaming

### Typical Workflow Patterns

#### Pattern 1: Web Application Thread Management

```
Client Application:
1. REST: GET /api/v1/threads → Display thread list
2. User clicks "Resume Thread"
3. REST: POST /api/v1/threads/{id}/resume → Resume thread
4. WebSocket: Connect → Real-time streaming of resumed thread
5. WebSocket: Send input, receive events
6. User navigates away → Disconnect WebSocket
7. REST: GET /api/v1/threads → Refresh list
```

#### Pattern 2: File Upload + Processing

```
Client Application:
1. REST: POST /api/v1/files/upload → Upload document
2. REST: POST /api/v1/threads → Create new thread
3. WebSocket: Connect → Stream processing
4. WebSocket: Send input with file reference
5. WebSocket: Receive streaming events (file analysis)
6. REST: GET /api/v1/threads/{id}/artifacts → List generated artifacts
7. REST: GET /api/v1/files/{artifact_id} → Download result
```

#### Pattern 3: Monitoring Dashboard

```
Client Application:
1. REST: GET /api/v1/status → Display daemon status
2. REST: GET /api/v1/health → Health check
3. REST: GET /api/v1/threads → List all threads
```

## Security Requirements

### Transport Security

#### Unix Socket

- **Default permissions**: `0o600` (owner read/write only)
- **Location**: `$SOOTHE_HOME/soothe.sock` (user home directory)
- **Threat model**: Local privilege escalation
- **Mitigation**: Filesystem permissions + PID lock prevents multiple daemons

#### WebSocket Localhost

- **Default bind**: `127.0.0.1:8765` (not externally accessible)
- **CORS**: Validate `Origin` header against allowed patterns
- **Threat model**: Malicious local website connecting to daemon
- **Mitigation**: CORS validation

#### WebSocket and HTTP REST Production

- **Recommended deployment**: Use reverse proxy (nginx, Caddy, Traefik)
- **Reverse proxy handles**:
  - TLS termination (HTTPS/WSS)
  - Authentication (API keys, JWT, OAuth)
  - Rate limiting
  - CORS validation
  - IP whitelisting
  - Request logging
- **Threat model**: Network eavesdropping, unauthorized access
- **Mitigation**: Reverse proxy handles security, Soothe focuses on orchestration

### CORS Configuration

**Allowed Origins**:
- Pattern matching using glob syntax (e.g., `http://localhost:*`)
- Validate `Origin` header against allowed patterns
- Reject connections with invalid origins

**Default Allowed Patterns**:
- `http://localhost:*`
- `http://127.0.0.1:*`
- Custom domains (e.g., `https://soothe.example.com`)

### Deployment Patterns

#### Pattern 1: Local Development (No Auth)

```
Client → Unix Socket/WebSocket → Soothe Daemon
```

- Suitable for local development and trusted environments
- No authentication required
- Filesystem permissions provide security

#### Pattern 2: Production Deployment (Auth via Reverse Proxy)

```
Client → Reverse Proxy (Auth + TLS) → Soothe Daemon
         ↓
    Validate API key/JWT
    Enforce rate limits
    Handle CORS
    Terminate TLS
```

- Reverse proxy handles all security concerns
- Soothe daemon trusts the reverse proxy
- Clear separation of concerns

#### Pattern 3: Desktop Application (Tauri/Electron)

```
Desktop App → Local WebSocket → Soothe Daemon
```

- Local WebSocket connection (127.0.0.1)
- No network exposure
- Optional: Desktop app manages user authentication separately

### Input Validation

**Message Size Limits**:
```python

```

**Schema Validation**:
```python
from typing import Any

def validate_message(msg: dict[str, Any]) -> list[str]:
    """Validate message structure. Returns list of errors."""
    errors = []

    if "type" not in msg:
        errors.append("Missing required field: type")
        return errors

    msg_type = msg["type"]

    if msg_type == "input":
        if "text" not in msg:
            errors.append("Input message missing required field: text")
        if not isinstance(msg.get("text"), str):
            errors.append("Input text must be a string")

    elif msg_type == "command":
        if "cmd" not in msg:
            errors.append("Command message missing required field: cmd")

    elif msg_type == "resume_thread":
        if "thread_id" not in msg:
            errors.append("Resume thread message missing required field: thread_id")

    # ... other message types

    return errors
```

## Naming Conventions

- **Unix socket**: Referenced as `"unix_socket"` in logs and metrics

## Breaking Changes

### Version 2.0 Protocol Changes

**This RFC introduces BREAKING CHANGES to the daemon protocol.** All existing clients must be updated.

#### Mandatory Thread Subscription

**Before (v1.x)**:
- All clients received all events via global broadcast
- No subscription mechanism
- Events mixed when multiple clients connected

**After (v2.0)**:
- Clients MUST send `subscribe_thread` message to receive events
- No global broadcast - only subscribed clients receive events
- Complete client isolation between threads

#### Client Migration Required

**TUI Client** (`src/soothe/ux/tui/app.py`):
```python
# OLD: Just connect
await client.connect()

# NEW: Connect + subscribe
await client.connect()
await client.send_new_thread()  # or send_resume_thread(thread_id)
thread_id = await client.receive_thread_id()  # from status message
await client.subscribe_thread(thread_id)  # NEW: required
```

**CLI Headless** (`src/soothe/ux/cli/execution/daemon_runner.py`):
```python
# OLD: Just send input
await client.send_input(text)

# NEW: Subscribe first
await client.connect()
await client.send_new_thread()
thread_id = await client.receive_thread_id()
await client.subscribe_thread(thread_id)  # NEW: required
await client.send_input(text)
```

**WebSocket Clients**:
```javascript
// OLD: Just connect and listen
const ws = new WebSocket('ws://localhost:8765');
ws.onmessage = (event) => console.log(event.data);

// NEW: Subscribe after connection
const ws = new WebSocket('ws://localhost:8765');
ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === 'status') {
    // Subscribe to thread
    ws.send(JSON.stringify({
      type: 'subscribe_thread',
      thread_id: msg.thread_id
    }));
  }
};
```

#### Event Message Changes

**Before (v1.x)**:
```json
{
  "type": "event",
  "namespace": [],
  "mode": "messages",
  "data": [...]
}
```

**After (v2.0)**:
```json
{
  "type": "event",
  "thread_id": "abc123",  // NEW: required for routing
  "namespace": [],
  "mode": "messages",
  "data": [...]
}
```

#### Status Message Changes

**Before (v1.x)**:
```json
{
  "type": "status",
  "state": "running",
  "thread_id": "abc123"
}
```

**After (v2.0)**:
```json
{
  "type": "status",
  "state": "running",
  "thread_id": "abc123",
  "client_id": "client-uuid-12345"  // NEW: unique per client
}
```

### Migration Strategy

**Hard-Cut Migration** - No backward compatibility:

1. **Remove old broadcast code** - Delete legacy `_broadcast()` fan-out logic
2. **Update all clients atomically** - TUI, CLI, WebSocket in single PR
3. **Require subscription** - Reject events without subscription
4. **Clean break** - No gradual migration, no fallback

**Rationale**:
- Maintaining backward compatibility adds complexity
- Event mixing is a security and correctness issue
- Clean architecture preferred over transitional code
- Single atomic update ensures consistency
