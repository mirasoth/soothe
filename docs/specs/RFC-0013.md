# RFC-0013: Unified Daemon Communication Protocol

**RFC**: 0013
**Title**: Unified Daemon Communication Protocol for Multi-Transport IPC
**Status**: Draft
**Kind**: Architecture Design
**Created**: 2026-03-19
**Dependencies**: RFC-0001, RFC-0002, RFC-0003

## Abstract

This RFC defines a transport-agnostic daemon communication protocol that supports both Unix domain sockets and WebSockets. The protocol specifies a common JSON-based message format, authentication model, security requirements, and implementation interface. This architecture enables the Soothe daemon to serve both local CLI/TUI clients (via Unix socket) and remote/web clients (via WebSocket) using the same protocol layer, while maintaining backward compatibility with existing clients.

## Motivation

### Problem: Single Transport Limitation

RFC-0003 defines the daemon IPC architecture using Unix domain sockets exclusively. This design has limitations:

1. **Browser incompatibility**: Web browsers cannot connect to Unix sockets, preventing web-based UI development
2. **Remote access**: Unix sockets are local-only, requiring SSH tunneling or local presence for remote access
3. **Mobile clients**: Mobile applications cannot connect to Unix sockets
4. **Infrastructure constraints**: Cloud deployments and containerized environments may prefer HTTP-based protocols

### Problem: No Protocol Specification

The current implementation in `src/soothe/cli/daemon/protocol.py` provides encoding/decoding utilities but lacks:
- Formal message schema definitions
- Transport abstraction layer
- Authentication specification
- Security requirements for different transport modes
- Versioning and compatibility guidelines

### Design Goals

1. **Transport independence**: Same message protocol over Unix socket and WebSocket
2. **Backward compatibility**: Existing Unix socket clients continue working unchanged
3. **Security by default**: Authentication required for remote access, optional for local
4. **Protocol evolution**: Versioning and extension mechanisms
5. **Clear specification**: Formal message schemas and implementation requirements

## Guiding Principles

### Principle 1: Protocol-Transport Separation

The message protocol (what we send) is independent of the transport mechanism (how we send it). Both Unix socket and WebSocket use the same JSON message format, with transport-specific framing.

### Principle 2: Security Tiers

Different transport modes have different security requirements:
- **Unix socket**: Filesystem permissions provide local security, no additional authentication needed
- **WebSocket localhost**: Optional authentication, suitable for local web UIs
- **WebSocket remote**: Mandatory authentication, TLS required, strict CORS

### Principle 3: Minimal Wire Overhead

The protocol uses JSON for simplicity and debuggability, not binary formats. Message framing is minimal:
- Unix socket: newline-delimited JSON lines
- WebSocket: native text frames (no newline needed)

### Principle 4: Streaming-First Design

The protocol is designed for bidirectional streaming, not request-response. Clients and servers exchange asynchronous messages over persistent connections.

### Component Responsibilities

#### Unix Socket Server

**Responsibility**: Serve local CLI/TUI clients over Unix domain socket.

**Capabilities**:
- Path: `$SOOTHE_HOME/soothe.sock` (default: `~/.soothe/soothe.sock`)
- Authentication: None (relies on filesystem permissions)
- Security: OS-level file permissions (user-only by default)
- Performance: ~0.1ms latency, ~1GB/s throughput
- Backward compatibility: Maintains existing behavior from RFC-0003

#### WebSocket Server

**Responsibility**: Serve web-based and remote clients over WebSocket protocol.

**Capabilities**:
- Default host: `127.0.0.1` (localhost only by default)
- Default port: `8765`
- Authentication: Optional for localhost, mandatory for remote
- Security: TLS support, CORS validation, rate limiting
- Performance: ~0.5ms latency (localhost), ~10-50ms (remote)

#### Protocol Handler

**Responsibility**: Transport-agnostic message processing.

**Capabilities**:
- Message validation (schema checking)
- Authentication enforcement (when required)
- Message routing to appropriate handlers
- Broadcast to all connected clients
- Client lifecycle management

#### Message Router

**Responsibility**: Route messages to SootheRunner and broadcast events.

**Capabilities**:
- Route client inputs to runner
- Stream runner events to clients
- Handle slash commands
- Thread management operations

### Architectural Constraints

1. **Single daemon instance**: Only one daemon can run per user (PID lock enforcement)
2. **Unified message schema**: All transports use identical JSON message format
3. **Transport abstraction**: Protocol layer has no transport-specific logic
4. **Authentication tiered**: Security requirements vary by transport mode
5. **Backward compatible**: Unix socket behavior unchanged from RFC-0003
6. **No message correlation**: Protocol does not use request IDs (streaming design)

## Protocol Specification

### Message Format

All messages are JSON objects with a required `type` field.

#### Base Message Schema

```json
{
  "type": "string (required)",
  "...": "additional fields based on type"
}
```

### Client → Server Messages

#### Input Message

Send user input to the daemon for processing.

```json
{
  "type": "input",
  "text": "string (required)",
  "autonomous": "boolean (optional, default: false)",
  "max_iterations": "integer (optional, default: null)"
}
```

**Example**:
```json
{
  "type": "input",
  "text": "Analyze the authentication code",
  "autonomous": false
}
```

#### Command Message

Send a slash command to the daemon.

```json
{
  "type": "command",
  "cmd": "string (required)"
}
```

**Example**:
```json
{
  "type": "command",
  "cmd": "/help"
}
```

**Valid commands**:
- `/help`, `/exit`, `/quit`, `/detach`
- `/plan`, `/memory`, `/context`, `/policy`, `/history`, `/review`
- `/thread list`, `/thread resume <id>`, `/thread archive <id>`
- `/clear`, `/config`, `/thread`

#### Resume Thread Message

Request the daemon to resume a specific conversation thread.

```json
{
  "type": "resume_thread",
  "thread_id": "string (required)"
}
```

**Example**:
```json
{
  "type": "resume_thread",
  "thread_id": "abc123-def456-ghi789"
}
```

#### Detach Message

Notify the daemon that the client is detaching (daemon keeps running).

```json
{
  "type": "detach"
}
```

#### Authentication Message (WebSocket Only)

Authenticate with the daemon (required for remote WebSocket, optional for localhost).

```json
{
  "type": "auth",
  "token": "string (required)"
}
```

**Example (API Key)**:
```json
{
  "type": "auth",
  "token": "sk_live_abc123def456"
}
```

**Example (JWT)**:
```json
{
  "type": "auth",
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

### Server → Client Messages

#### Status Message

Daemon state change notification (sent on connect and state transitions).

```json
{
  "type": "status",
  "state": "string (required: idle|running|stopped|stopping|detached)",
  "thread_id": "string (required, may be empty)",
  "input_history": "array of strings (optional, last 100 entries)"
}
```

**Example**:
```json
{
  "type": "status",
  "state": "running",
  "thread_id": "abc123",
  "input_history": ["query 1", "query 2", "query 3"]
}
```

**States**:
- `idle`: Daemon is ready and waiting for input
- `running`: Daemon is processing a query
- `stopped`: Daemon is shutting down
- `stopping`: Daemon received stop signal
- `detached`: Client has detached (daemon still running)

#### Event Message

Stream event from SootheRunner (canonical `(namespace, mode, data)` format).

```json
{
  "type": "event",
  "namespace": "array of strings (required)",
  "mode": "string (required: messages|updates|custom)",
  "data": "any (required, based on mode)"
}
```

**Example (messages mode)**:
```json
{
  "type": "event",
  "namespace": [],
  "mode": "messages",
  "data": ["AIMessage content...", {"langgraph_metadata": {}}]
}
```

**Example (custom mode)**:
```json
{
  "type": "event",
  "namespace": [],
  "mode": "custom",
  "data": {
    "type": "soothe.plan.step_started",
    "step_id": "step_1",
    "description": "Analyze codebase structure"
  }
}
```

**Event Types**: See RFC-0003 "Stream Events" section for complete event taxonomy.

#### Command Response Message

Output from slash command execution.

```json
{
  "type": "command_response",
  "content": "string (required)"
}
```

**Example**:
```json
{
  "type": "command_response",
  "content": "Available commands:\n/help - Show commands\n/exit - Exit TUI\n..."
}
```

#### Authentication Response Message (WebSocket Only)

Response to authentication attempt.

```json
{
  "type": "auth_response",
  "success": "boolean (required)",
  "error": "string (optional, present if success=false)",
  "client_id": "string (optional, unique identifier if authenticated)"
}
```

**Example (Success)**:
```json
{
  "type": "auth_response",
  "success": true,
  "client_id": "client_abc123"
}
```

**Example (Failure)**:
```json
{
  "type": "auth_response",
  "success": false,
  "error": "Invalid API key"
}
```

#### Error Message

Protocol error notification.

```json
{
  "type": "error",
  "code": "string (required)",
  "message": "string (required)",
  "details": "object (optional)"
}
```

**Example**:
```json
{
  "type": "error",
  "code": "INVALID_MESSAGE",
  "message": "Message missing required field: type",
  "details": {"received": {"text": "hello"}}
}
```

**Error Codes**:
- `INVALID_MESSAGE`: Malformed JSON or missing required fields
- `AUTHENTICATION_REQUIRED`: Auth needed but not provided
- `AUTHENTICATION_FAILED`: Invalid credentials
- `RATE_LIMITED`: Too many messages
- `INTERNAL_ERROR`: Server-side error

## Transport Layer Specification

### Unix Domain Socket

**Wire Format**: Newline-delimited JSON lines (JSONL)

**Message Framing**:
```
<json-message>\n
```

**Connection Lifecycle**:
1. Client connects to `$SOOTHE_HOME/soothe.sock`
2. Server immediately sends initial `status` message
3. Client sends messages, each terminated with `\n`
4. Server sends messages, each terminated with `\n`
5. Connection closes on EOF or error

**Implementation**:
```python
# Server side
reader, writer = await asyncio.open_unix_connection(path=str(socket_path))
message_bytes = encode(message)  # Returns: json.dumps(msg) + "\n"
writer.write(message_bytes)
await writer.drain()

# Client side
line = await reader.readline()  # Reads until \n
message = decode(line)  # Returns: json.loads(line)
```

### WebSocket

**Wire Format**: WebSocket text frames (RFC 6455)

**Message Framing**:
```
WebSocket Frame:
  - opcode: 0x01 (text)
  - payload: <json-message>  (no newline needed)
```

**Connection Lifecycle**:
1. Client initiates WebSocket handshake at `ws://host:port` or `wss://host:port`
2. Server accepts handshake, optionally validates CORS origin
3. **If auth required**: Client must send `auth` message within 5 seconds
4. Server sends initial `status` message (or `auth_response` if auth required)
5. Client and server exchange JSON messages in text frames
6. Connection closes with WebSocket close frame (code 1000 for normal, 1008 for auth failure)

**Implementation**:
```python
# Server side (using websockets library)

                return

            await websocket.close(code=1008, reason="Authentication timeout")
            return

    # Send initial status
    await websocket.send(json.dumps(initial_status))

    # Message loop
    async for message in websocket:
        msg = json.loads(message)
        # Process message...

# Client side

# Send auth if required
```

## Authentication Specification

### Authentication Modes

#### Mode 1: Disabled (Unix Socket Default)

**Applicable to**: Unix socket only

**Behavior**:
- No authentication messages exchanged
- Relies on filesystem permissions for security
- Socket file has mode `0o644` (user read/write only by default)

**Implementation**:
```python
# Unix socket authentication always succeeds
auth_context = AuthContext(authenticated=True, mode="unix_socket", client_id="local")
```

#### Mode 2: API Key

**Applicable to**: WebSocket (localhost and remote)

**Behavior**:
- Client sends `{"type": "auth", "token": "<api_key>"}`
- Server validates against `~/.soothe/api_keys.json`
- API keys are static tokens with descriptions and permissions

**API Key Storage** (`~/.soothe/api_keys.json`):
```json
{
  "version": 1,
  "keys": [
    {
      "id": "key_001",
      "token": "sk_live_abc123def456ghi789",
      "description": "Web UI",
      "created_at": "2026-03-19T00:00:00Z",
      "last_used_at": "2026-03-19T12:00:00Z",
      "permissions": ["read", "write"]
    }
  ]
}
```

**Validation**:
```python
def validate_api_key(token: str, keys_file: Path) -> dict | None:
    """Validate API key and return key metadata."""
    try:
        data = json.loads(keys_file.read_text())
        for key in data.get("keys", []):
            if key.get("token") == token:
                # Update last_used_at
                key["last_used_at"] = datetime.now(timezone.utc).isoformat()
                keys_file.write_text(json.dumps(data, indent=2))
                return {"key_id": key["id"], "permissions": key["permissions"]}
    except Exception:
        pass
    return None
```

**CLI Commands** (Planned):
```bash

```

#### Mode 3: JWT

**Applicable to**: WebSocket (remote, production)

**Behavior**:
- Client sends `{"type": "auth", "token": "<jwt_token>"}`
- Server validates JWT signature, expiry, and claims
- Supports third-party identity providers (future)

**JWT Claims**:
```json
{
  "sub": "user_id",
  "iat": 1234567890,
  "exp": 1234657890,
  "permissions": ["read", "write"]
}
```

**Validation**:
```python
import jwt

def validate_jwt(token: str, secret: str, algorithm: str = "HS256") -> dict | None:
    """Validate JWT token and return claims."""
    try:
        claims = jwt.decode(token, secret, algorithms=[algorithm])
        if datetime.now(timezone.utc).timestamp() > claims.get("exp", 0):
            return None
        return claims
    except jwt.InvalidTokenError:
        return None
```

### Authentication Enforcement

**Decision Matrix**:

| Transport | Config | Authentication Required? |
|-----------|--------|--------------------------|
| Unix socket | Any | No (filesystem permissions) |
| WebSocket localhost | `require_for_localhost=false` | No |
| WebSocket localhost | `require_for_localhost=true` | Yes |
| WebSocket remote | Any | Yes (mandatory) |
| WebSocket remote + TLS | Any | Yes (mandatory) |

**Implementation**:
```python
def requires_authentication(
    transport: TransportMode,
    remote_addr: str | None,
    config: DaemonAuthConfig
) -> bool:
    """Determine if authentication is required for this connection."""
    if transport == TransportMode.UNIX_SOCKET:
        return False

    # WebSocket
    is_localhost = remote_addr in ("127.0.0.1", "::1", "localhost")
    if is_localhost and not config.require_for_localhost:
        return False

    return config.enabled
```

## Security Requirements

### Transport Security

#### Unix Socket

- **Default permissions**: `0o600` (owner read/write only)
- **Location**: `$SOOTHE_HOME/soothe.sock` (user home directory)
- **Threat model**: Local privilege escalation
- **Mitigation**: Filesystem permissions + PID lock prevents multiple daemons

#### WebSocket Localhost

- **Default bind**: `127.0.0.1:8765` (not externally accessible)
- **CORS**: Validate `Origin` header against allowed patterns
- **Threat model**: Malicious local website connecting to daemon
- **Mitigation**: CORS validation + optional authentication

#### WebSocket Remote

- **Requirement**: TLS mandatory (`wss://`)
- **Certificate**: Valid TLS certificate (Let's Encrypt or self-signed for dev)
- **CORS**: Strict origin validation
- **Threat model**: Network eavesdropping, MITM, replay attacks
- **Mitigation**: TLS + authentication + rate limiting

### CORS Configuration

```python
import fnmatch

ALLOWED_ORIGINS = [
    "http://localhost:*",
    "http://127.0.0.1:*",
    "https://soothe.example.com"
]

def check_origin(origin: str, allowed: list[str]) -> bool:
    """Check if origin is allowed."""
    for pattern in allowed:
        if fnmatch.fnmatch(origin, pattern):
            return True
    return False
```

### Rate Limiting

**Configuration**:
```python
class RateLimitConfig(BaseModel):
    enabled: bool = True
    max_messages_per_second: int = 10
    max_connections_per_ip: int = 5
    max_total_connections: int = 100
```

**Implementation** (Token Bucket):
```python
import asyncio
from collections import defaultdict

class RateLimiter:
    def __init__(self, rate: int = 10):
        self.rate = rate  # messages per second
        self.buckets: dict[str, float] = defaultdict(lambda: rate)
        self.last_update: dict[str, float] = defaultdict(lambda: time.time())

    async def check(self, client_id: str) -> bool:
        """Check if client can send message (non-blocking)."""
        now = time.time()
        elapsed = now - self.last_update[client_id]

        # Refill tokens
        self.buckets[client_id] = min(
            self.rate,
            self.buckets[client_id] + elapsed * self.rate
        )
        self.last_update[client_id] = now

        # Check and consume
        if self.buckets[client_id] >= 1.0:
            self.buckets[client_id] -= 1.0
            return True
        return False
```

```python
def validate_message(msg: dict[str, Any]) -> list[str]:
```

## Naming Conventions

### Message Types

- **Format**: `snake_case` (e.g., `input`, `command`, `resume_thread`)
- **Namespace**: Client-to-server messages use simple nouns; server-to-client use descriptive names

### Event Types

- **Format**: `soothe.<component>.<action>` (e.g., `soothe.plan.step_started`)
- **Component**: Protocol name or subagent name
- **Action**: Past tense for completed actions (e.g., `projected`, `completed`), present for ongoing (e.g., `started`)

### Transport Identifiers

- **Unix socket**: Referenced as `"unix_socket"` in logs and metrics
- **WebSocket**: Referenced as `"websocket"` with connection metadata (remote_addr, origin)

## Error Handling

### Protocol Errors

**Invalid JSON**:
```json
{
  "type": "error",
  "code": "INVALID_JSON",
  "message": "Failed to parse message as JSON",
  "details": {"raw": "not valid json"}
}
```

**Missing Required Field**:
```json
{
  "type": "error",
  "code": "INVALID_MESSAGE",
  "message": "Message missing required field: type",
  "details": {"received": {"text": "hello"}}
}
```

**Unknown Message Type**:
```json
{
  "type": "error",
  "code": "UNKNOWN_MESSAGE_TYPE",
  "message": "Unknown message type: unknown_type",
  "details": {"type": "unknown_type"}
}
```

### Authentication Errors

**Authentication Required**:
```json
{
  "type": "error",
  "code": "AUTHENTICATION_REQUIRED",
  "message": "Authentication required for this connection",
  "details": {"transport": "websocket", "remote_addr": "192.168.1.100"}
}
```

**Authentication Failed**:
```json
{
  "type": "auth_response",
  "success": false,
  "error": "Invalid API key"
}
```

### Rate Limiting Errors

```json
{
  "type": "error",
  "code": "RATE_LIMITED",
  "message": "Rate limit exceeded: max 10 messages per second",
  "details": {
    "limit": 10,
    "window_seconds": 1,
    "retry_after_ms": 100
  }
}
```

## Examples

### Example 1: Unix Socket Connection

```
Client connects to ~/.soothe/soothe.sock
→ Server sends: {"type":"status","state":"idle","thread_id":"","input_history":[]}

Client sends: {"type":"input","text":"hello"}
← Server receives, processes

Server sends: {"type":"status","state":"running","thread_id":"abc123"}
Server streams: {"type":"event","namespace":[],"mode":"messages","data":[...]}
Server streams: {"type":"event","namespace":[],"mode":"custom","data":{"type":"soothe.plan.step_started",...}}
Server sends: {"type":"status","state":"idle","thread_id":"abc123"}
```

### Example 2: WebSocket Connection (Localhost, No Auth)

```
Client connects to ws://localhost:8765
→ Server sends: {"type":"status","state":"idle","thread_id":"","input_history":[]}

Client sends: {"type":"input","text":"analyze code"}
← Server processes, streams events

Server streams: {"type":"event",...}
Server sends: {"type":"status","state":"idle","thread_id":"xyz789"}
```

### Example 3: WebSocket Connection (Remote, With Auth)

```
Client connects to wss://soothe.example.com:8765
Origin: https://app.example.com
← Server validates CORS origin (success)

Client sends: {"type":"auth","token":"sk_live_abc123"}
→ Server validates API key
→ Server sends: {"type":"auth_response","success":true,"client_id":"client_001"}

Server sends: {"type":"status","state":"idle","thread_id":"","input_history":[]}

Client sends: {"type":"input","text":"research quantum computing"}
← Server processes, streams events
```

### Example 4: Authentication Failure

```
Client connects to wss://soothe.example.com:8765

Client sends: {"type":"auth","token":"invalid_key"}
→ Server validates (failure)
→ Server sends: {"type":"auth_response","success":false,"error":"Invalid API key"}
→ Server closes connection (WebSocket close code 1008)
```

### Example 5: Rate Limiting

```
Client sends 15 messages in 1 second (limit: 10/sec)

On 11th message:
→ Server sends: {"type":"error","code":"RATE_LIMITED","message":"Rate limit exceeded",...}
→ Server closes connection (optional, based on config)
```

## Migration Path

### Phase 1: Protocol Abstraction (Week 1)

**Goal**: Create transport-agnostic protocol layer.

**Changes**:
1. Create `src/soothe/cli/daemon/protocol_v2.py` with abstract `MessageProtocol` interface
2. Refactor Unix socket to use `UnixSocketProtocol` implementation
3. Add message validation and error handling
4. Add configuration schema for transports

**Backward Compatibility**: 100% - existing clients unchanged

### Phase 2: WebSocket Implementation (Week 2)

**Goal**: Add WebSocket server alongside Unix socket.

**Changes**:
1. Add `websockets>=12.0` to dependencies
2. Implement WebSocket server in `SootheDaemon.start()`
3. Implement `WebSocketProtocol` class
4. Both protocols share same message handling logic
5. Update client to support WebSocket connections

**Backward Compatibility**: 100% - Unix socket behavior unchanged

### Phase 3: Authentication System (Week 3)

**Goal**: Add authentication for WebSocket.

**Changes**:
1. Implement API key storage and validation
2. Add authentication flow to WebSocket handshake
3. Add CLI commands: `soothe auth create-key`, `soothe auth list-keys`, `soothe auth revoke-key`
4. Add rate limiting
5. Add CORS validation

**Backward Compatibility**: 100% - Unix socket remains auth-free, WebSocket auth is optional for localhost

### Phase 4: Documentation & Testing (Week 4)

**Goal**: Complete documentation and test coverage.

**Changes**:
1. Add comprehensive unit tests for WebSocket protocol
2. Add integration tests for dual-protocol operation
3. Update configuration documentation
4. Create migration guide for users
