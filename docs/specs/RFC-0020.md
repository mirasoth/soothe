# RFC-0020: Event Display Architecture

**RFC**: 0020
**Title**: Event Display Architecture
**Status**: Draft
**Kind**: Architecture Design
**Created**: 2026-03-26
**Updated**: 2026-03-26
**Dependencies**: RFC-0001, RFC-0002, RFC-0013, RFC-0015, RFC-0019

## Abstract

This RFC establishes the architectural foundation for displaying agent activity events across CLI and TUI interfaces. The design introduces a **registry-driven, two-level tree display system** that provides consistent, readable output while maintaining extensibility for new event types. By separating event registration from display logic, this architecture ensures that new capabilities automatically integrate into the display layer without requiring renderer modifications.

## Motivation

### Problem: Fragmented Event Display

Current event rendering suffers from three critical issues:

1. **Inconsistent Visual Hierarchy**: Tools, subagents, and protocol events use different display patterns, creating cognitive load for users
2. **Manual Display Logic**: Renderers hardcode string formatting for each event type, requiring code changes for every new event
3. **No Progressive Disclosure**: All events appear at the same verbosity level, overwhelming users with protocol internals

### Problem: Tight Coupling

The current implementation requires:
- Renderer modification for every new event type
- Duplicate string formatting between event definition and display
- Manual synchronization of display patterns across CLI and TUI

### Solution: Registry-Driven Display

This RFC introduces a **registry-driven architecture** where:
- Events register their display metadata (templates, verbosity) at definition time
- Renderers query the registry for display information
- Display logic is driven by configuration, not code

This creates a **separation of concerns**:
- **Event authors** define display metadata alongside event types
- **Renderer implementers** query the registry for consistent display behavior
- **Display patterns** emerge from metadata conventions, not hardcoded logic

## Design Principles

### Principle 1: Registry-Driven Display Metadata

**Rule**: All display-relevant event metadata must be registered at event definition time.

**Why**: Eliminates the need for renderer code changes when adding new events.

**What This Means**:
- Event types register `summary_template` strings with field interpolation
- Event types declare `verbosity` categories for visibility filtering
- Renderers query the registry for templates, never hardcode strings

**Example Registration**:
```
Event: BrowserStepEvent
  - type: "soothe.subagent.browser.step"
  - summary_template: "Step {step}"
  - verbosity: "subagent_progress"
```

### Principle 2: Two-Level Tree Structure

**Rule**: All activity events display as a two-level tree: summary + details.

**Why**: Provides immediate visual context while respecting terminal width constraints.

**Display Pattern**:
```
Level 1 (Summary):  ● EventSummary
Level 2 (Details):    └ Additional context or results
```

**When Details Appear**:
- Tool results show completion status or error
- Subagent steps show action and URL
- Text events show content preview
- Result events show cost and duration

**When Details Are Omitted**:
- Simple status events with no additional context
- Events where summary already contains all relevant information

### Principle 3: Progressive Disclosure via Verbosity

**Rule**: Event visibility is controlled by verbosity categories, not hardcoded filters.

**Verbosity Categories**:
- `assistant_text`: AI responses (visible in normal mode)
- `protocol`: Core agent activity (visible in normal mode)
- `subagent_progress`: Subagent step completion (visible in normal mode)
- `tool_activity`: Granular tool execution (hidden in normal mode)
- `debug`: Internal protocol details (hidden in normal mode)

**Display Rules**:
- Normal mode: `assistant_text`, `protocol`, `subagent_progress`, `error`
- Verbose mode: Adds `tool_activity`
- Debug mode: All categories

### Principle 4: Template Interpolation

**Rule**: Event summaries use template strings with field interpolation, not manual formatting.

**Template Syntax**: Python format strings with field names from event data.

**Why**: Templates are declarative, portable, and extensible.

**Examples**:
- `"Step {step}"` → "Step 1"
- `"Tool: {tool}"` → "Tool: read_file"
- `"Done (${cost_usd}, {duration_ms}ms)"` → "Done ($0.0023, 1234ms)"

### Principle 5: Visual Consistency via Display Helpers

**Rule**: All display formatting uses shared helper functions for visual consistency.

**Required Helpers**:
- `make_dot_line(color, summary, details)` → Two-level tree output
- `make_tool_block(name, args, result, status)` → Tool call display
- `format_event_summary(event_type, data, registry)` → Registry-driven summary

**Why**: Ensures consistent indentation, colors, and icons across all renderers.

## Architecture Overview

### Component Layers

```
┌─────────────────────────────────────────────────────────────┐
│  Event Definition Layer                                      │
│  - Event types with Pydantic models                          │
│  - Registry metadata (template, verbosity)                   │
│  - Self-registration at module load time                     │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  Event Registry (event_catalog.py)                           │
│  - Global REGISTRY singleton                                 │
│  - EventMeta storage: {type, template, verbosity}           │
│  - O(1) lookup by event type string                          │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  Event Dispatcher (event_processor.py)                       │
│  - Filter by verbosity category                              │
│  - Classify events (assistant_text, protocol, etc.)         │
│  - Route to appropriate renderer                             │
└──────────────────────┬──────────────────────────────────────┘
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
┌──────────────────┐       ┌──────────────────┐
│  CLI Renderer    │       │  TUI Renderer    │
│  (Headless)      │       │  (Interactive)   │
│                  │       │                  │
│  - Query registry│       │  - Query registry│
│  - Tree display  │       │  - Tree display  │
│  - ANSI colors   │       │  - Rich widgets  │
└──────────────────┘       └──────────────────┘
```

### Separation of Concerns

**Event Definition Layer** (Author responsibility):
- Define event type string and Pydantic model
- Register summary template with field names
- Declare verbosity category
- Emit events with properly typed data

**Event Registry** (Infrastructure):
- Store and index event metadata
- Provide O(1) lookup by event type
- Validate template syntax at registration time
- Supply default verbosity based on event domain

**Event Dispatcher** (Orchestration):
- Classify events by category
- Apply verbosity filters
- Route to CLI or TUI renderer
- Handle event streams (messages, updates, custom)

**Renderer Layer** (Display logic):
- Query registry for templates
- Format summaries using template interpolation
- Extract additional details for second-level display
- Apply visual styling (colors, icons, indentation)

## Event Registration Requirements

### Required Metadata

Every event type must register:

**1. Event Type String**: Unique identifier in `soothe.<domain>.<component>.<action>` format

**2. Summary Template**: Format string with field interpolation
- Must reference fields present in event data
- Should be concise (recommend < 50 characters)
- Must not reference nested fields (only top-level event fields)

**3. Verbosity Category**: One of the standard categories
- If not specified, defaults based on event domain

### Registration Pattern

Events register themselves at module load time:

```
1. Define Pydantic model with 'type' field default
2. Call register_event() with:
   - Event class
   - verbosity (optional, defaults based on domain)
   - summary_template (required for visible events)
3. Event automatically available to all renderers
```

### Template Validation

The registry validates templates at registration time:
- Template must use valid Python format string syntax
- Referenced fields must exist in event model
- Template formatting errors are logged (not raised) to prevent runtime crashes

## Display Implementation Requirements

### Summary Building

Renderers must implement summary building as:

```
1. Query REGISTRY.get_meta(event_type)
2. If meta.summary_template exists:
   - Format with event data: template.format(**data)
   - Handle formatting errors gracefully
3. If no template, event is not displayed (or use fallback)
```

**Constraint**: Renderers must never hardcode event-specific string formatting.

### Detail Extraction

For two-level tree display, renderers implement detail extraction:

**Pattern**:
```
1. Identify event types that warrant second-level details
2. Extract relevant fields from event data
3. Format as concise string (recommend < 80 characters)
4. Pass to display helper as 'body' parameter
```

**Detail Categories**:
- **Tool results**: Completion status, error messages, result previews
- **Subagent steps**: Action description, URL, target information
- **Text content**: Preview of content (truncated to terminal width)
- **Result metrics**: Cost, duration, counts

**Constraint**: Detail extraction logic should be minimal and event-type-specific. This is the only place where renderer code may reference specific event types.

### Visual Consistency

All renderers must use shared display helpers:

**Two-Level Tree**:
- Use `make_dot_line(color, summary, details)` for consistent indentation
- Level 1: Colored bullet with summary text
- Level 2: Tree connector (└) with detail text

**Colors**:
- Assistant activity: `blue`
- Success/completion: `green`
- Error: `red`
- Progress/activity: `yellow`
- Subagent: `magenta`
- Protocol: `dim`

**Icons**:
- In progress: `●` (bullet)
- Success: `✓` (checkmark)
- Error: `✗` (X mark)
- Tool: `⚙` (gear)

## Verbosity Classification

### Category Definitions

**assistant_text**: AI-generated responses
- Examples: Text blocks, reasoning, final answers
- Visibility: Normal mode and above

**protocol**: Core agent lifecycle events
- Examples: Thread start/end, plan creation, checkpoint
- Visibility: Normal mode and above

**subagent_progress**: Subagent activity
- Examples: Browser steps, Claude tool use, subagent results
- Visibility: Normal mode and above

**tool_activity**: Detailed tool execution
- Examples: Tool call start, tool call complete
- Visibility: Verbose mode and above (hidden in normal)

**debug**: Internal protocol details
- Examples: Context projection, memory storage, durability
- Visibility: Debug mode only

### Domain Defaults

When verbosity is not explicitly set, the registry applies defaults:

- **lifecycle** domain → `protocol`
- **protocol** domain → `protocol`
- **subagent** domain → `subagent_progress`
- **tool** domain → `tool_activity`
- **output** domain → `assistant_text`
- **error** domain → `error` (always visible)

### Renderer Behavior

Renderers apply verbosity filtering before display:

```
1. Classify event by verbosity category (via registry lookup)
2. Check current verbosity level (normal/verbose/debug)
3. If event category is visible at current level:
   - Query registry for template
   - Format summary
   - Extract details
   - Display using helpers
4. If not visible, skip display
```

## Event Display Patterns

### Pattern: Tool Activity

**Registration**:
- Type: `soothe.tool.<tool_name>.call_started`
- Template: `<tool_name>({args_summary})`
- Verbosity: `tool_activity`

**Display**:
```
⚙ ToolName("key_argument")
  └ ✓ Result summary (duration)
```

**Note**: Tool activity is filtered at normal verbosity. Only subagent tools (browser, claude) with `subagent_progress` visibility appear in normal mode.

### Pattern: Subagent Progress

**Registration**:
- Type: `soothe.subagent.<name>.<action>`
- Template: Concise summary with key fields
- Verbosity: `subagent_progress`

**Display Examples**:

Browser step event:
```
● Step 1
  └ Navigate to page | https://example.com
```

Claude tool use event:
```
● Tool: read_file
```

Claude result event:
```
● Done ($0.0023, 1234ms)
```

**Visibility**: Shown in normal mode because `subagent_progress` is visible.

### Pattern: Protocol Events

**Registration**:
- Type: `soothe.protocol.<component>.<action>`
- Template: Concise status message
- Verbosity: `protocol`

**Display Example**:
```
● Thread xqrlpr212rs3 created
● Plan: Search for information (3 steps)
  └ ⚙ Step 1: Browse web
  └ ✓ Step 1 completed
```

**Visibility**: Shown in normal mode because `protocol` is visible.

### Pattern: Error Events

**Registration**:
- Type: `soothe.error.<component>.<type>`
- Template: Error description
- Verbosity: `error` (always visible)

**Display Example**:
```
✗ Tool execution failed
  └ compilation error at line 42
```

**Visibility**: Always shown regardless of verbosity level.

## Extensibility Model

### Adding New Event Types

To add a new event that displays correctly:

**1. Define Event Model** (in appropriate module):
```
class MyNewEvent(SubagentEvent):
    type: str = "soothe.subagent.my_feature.action"
    field1: str
    field2: int
```

**2. Register with Metadata**:
```
register_event(
    MyNewEvent,
    verbosity="subagent_progress",
    summary_template="Action: {field1} ({field2})",
)
```

**3. Emit Event**:
```
emit(MyNewEvent(field1="data", field2=42))
```

**Result**: Event automatically displays in CLI and TUI without renderer modifications.

### Adding New Verbosity Categories

To introduce a new visibility category:

**1. Define Category**: Add to `ProgressCategory` type in `progress_verbosity.py`

**2. Define Visibility Rules**: Update `should_show()` logic

**3. Register Events**: Use new category in `register_event()` calls

**4. Update Renderers**: No changes required (category is applied at filter level)

### Adding New Display Helpers

To introduce new visual patterns:

**1. Define Helper**: Create function in `ux/tui/utils.py` or `ux/cli/utils.py`

**2. Use Registry**: Helper should query registry for templates, not hardcode

**3. Document Pattern**: Add to this RFC under "Display Implementation Requirements"

**4. Apply to Renderers**: Update renderer to use new helper

## Terminal Formatting Standards

### Width Constraints

- Default terminal width: 80 characters
- Maximum summary length: 50 characters (template-driven)
- Maximum detail length: 80 characters
- Indentation for tree structure: 2 spaces + connector

### Text Preservation

All text processing must preserve word boundaries:
- Normalize whitespace to single spaces
- Ensure space between words after tag removal
- Truncate at word boundaries when possible
- Add ellipsis (...) for truncated text

### ANSI Color Support

- CLI renderer: Detect ANSI support, fallback to plain text
- TUI renderer: Always use Rich color system
- Color names: Use Rich color names (blue, green, red, yellow, magenta, dim)

### Error Handling

Display logic must be resilient:
- Template formatting errors → Log and skip display (never crash)
- Missing registry metadata → Skip display (invisible event)
- Malformed event data → Log warning and skip display
- Terminal width detection failure → Fall back to 80 chars

## Implementation Requirements

### Event Registration Phase

**Files**: All event definition modules (`*/events.py`)

**Requirements**:
1. All visible events must call `register_event()` with template
2. Verbosity must be explicitly set for non-default visibility
3. Templates must reference valid event fields
4. Registration happens at module load time (no manual registration calls)

### Registry Implementation

**File**: `src/soothe/core/event_catalog.py`

**Requirements**:
1. Provide global `REGISTRY` singleton
2. Support O(1) lookup by event type string
3. Validate template syntax at registration
4. Return `EventMeta` with `summary_template` and `verbosity`
5. Provide default verbosity based on event domain

### Renderer Implementation

**Files**: `src/soothe/ux/tui/renderer.py`, `src/soothe/ux/cli/renderer.py`

**Requirements**:
1. Query registry for all event summaries (never hardcode)
2. Implement detail extraction for two-level trees
3. Use shared display helpers (`make_dot_line`, etc.)
4. Apply verbosity filtering before display
5. Handle template formatting errors gracefully
6. Maintain visual consistency across CLI and TUI

### Testing Requirements

**Unit Tests**:
- Registry lookup for registered events
- Template formatting with valid data
- Template error handling with invalid data
- Verbosity filtering logic
- Detail extraction for different event types

**Integration Tests**:
- End-to-end event emission → display
- Registry-driven display for new events
- Two-level tree rendering
- Verbosity mode switching

**Manual Testing**:
- Visual consistency across CLI and TUI
- Terminal width edge cases
- ANSI color support/fallback
- Error event display

## Success Criteria

1. **Registry-Driven**: All event summaries use registry templates (no hardcoded strings in renderers)
2. **Consistent Display**: Tools, subagents, and protocol events follow the same two-level tree pattern
3. **Extensible**: New events display correctly by registering with template (zero renderer changes)
4. **Progressive Disclosure**: Verbosity filtering works correctly across all modes
5. **Resilient**: Template errors and missing metadata don't crash the display system
6. **Terminal-Optimized**: Output respects width constraints and preserves readability
7. **Visually Consistent**: Icons, colors, and indentation match across CLI and TUI

## Migration Path

For existing events without registry metadata:

**Phase 1**: Add `register_event()` calls to all existing event modules
- Add summary templates for user-visible events
- Set verbosity categories for visibility filtering

**Phase 2**: Update renderers to query registry
- Replace hardcoded string formatting with registry lookups
- Implement detail extraction for two-level trees

**Phase 3**: Remove legacy display logic
- Delete manual event handling code
- Consolidate all display logic to registry-driven approach

**Phase 4**: Verify and document
- Run full test suite
- Update user documentation
- Create examples for new event authors
