# RFC-0008: Request Processing Workflow and Performance Optimization

**RFC**: 0008
**Title**: Request Processing Workflow and Performance Optimization
**Status**: Draft
**Created**: 2026-03-16
**Updated**: 2026-03-16
**Related**: RFC-0001, RFC-0002, RFC-0003, RFC-0007

## Abstract

This RFC defines the architecture for Soothe's request processing workflow and establishes a framework for performance optimization. It specifies a multi-phase execution model (input processing, pre-stream, stream, post-stream) with intelligent query classification and adaptive resource allocation to minimize latency while maintaining quality.

## Motivation

RFC-0003 defines the CLI/TUI architecture and event streaming model. RFC-0007 adds autonomous iteration. However, neither RFC addresses performance optimization or defines how the system should adapt its processing based on query complexity.

### Problem Statement

Current behavior applies uniform heavyweight processing to all queries:
- **Memory recall** (1-2s): Semantic search over conversation history
- **Context projection** (0.5-1s): Token-budget allocation from context store
- **Plan creation** (2-3s): LLM-based planning via structured output

For trivial queries (e.g., "who is your father"), these operations waste 4-5 seconds while providing no value. This creates poor user experience for interactive use cases.

### Design Goals

1. **Sub-second response for trivial queries**: Greetings, simple questions, acknowledgments
2. **Efficient processing for simple queries**: File operations, basic searches, direct questions
3. **Full processing for complex queries**: Multi-step tasks, architecture decisions, refactoring
4. **No quality regression**: Complex tasks must retain full protocol capabilities
5. **Observable and configurable**: Performance metrics and feature flags for gradual rollout

## Design Principles

### 1. Adaptive Processing Based on Query Complexity

The system classifies queries into complexity levels and adjusts processing accordingly:

- **Trivial**: Greetings, simple questions, acknowledgments → Skip memory, context, LLM planning
- **Simple**: Direct operations, basic searches → Skip memory, context; use template planning
- **Medium**: Multi-step tasks requiring context → Full memory and context; template or LLM planning
- **Complex**: Architectural decisions, refactoring → Full protocol stack with LLM planning

### 2. Lazy Resource Loading

Resources are loaded only when needed:
- Memory recall is triggered only for queries that benefit from historical context
- Context projection is deferred until the query complexity warrants it
- Planning uses templates for common patterns, LLM only when necessary

### 3. Parallel Execution for Independent Operations

Independent operations execute concurrently:
- Memory recall and context projection run in parallel
- Embedding generation and database queries overlap
- Async I/O maximized throughout the pipeline

### 4. Caching at Multiple Levels

Strategic caching reduces redundant work:
- Embedding cache for similar queries
- Template cache for common plan patterns
- Context cache for repeated projections

### 5. Graceful Degradation

System remains functional even if optimization components fail:
- Classification failures default to "simple" (safe middle ground)
- Cache misses fall back to full computation
- Template misses fall back to LLM planning

## Architecture

### High-Level Request Processing Flow

```
User Input
    │
    ├─► Input Processing (< 10ms)
    │   ├─► Validate and normalize
    │   ├─► Slash command detection
    │   └─► Thread routing
    │
    └─► SootheRunner.astream()
        │
        ├─► Phase 0: Query Classification (< 1ms)
        │   └─► Determine complexity level
        │
        ├─► Phase 1: Pre-Stream (0-4.5s, depends on complexity)
        │   ├─► Thread Management (~12ms)
        │   ├─► Policy Check (~10ms)
        │   ├─► Memory Recall (0-2s, conditional)
        │   ├─► Context Projection (0-1s, conditional)
        │   └─► Plan Creation (0-3s, complexity-dependent)
        │
        ├─► Phase 2: Stream (varies by query)
        │   └─► LangGraph execution
        │
        └─► Phase 3: Post-Stream (< 100ms)
            ├─► Context Ingestion
            ├─► Memory Storage
            ├─► Plan Reflection
            └─► Thread Persistence
```

#### Heuristic Classification Rules

**Trivial** (target: < 10% of queries):
- Single-word greetings: "hi", "hello", "hey", "thanks"
- Simple questions with < 8 words: "who is your father?", "what time is it?"
- Acknowledgments: "ok", "yes", "no", "got it"

**Simple** (target: ~30% of queries):
- Direct operations: "read file.txt", "list files", "show config"
- Basic searches: "search for X", "find Y"
- Questions with 5-15 words

**Medium** (target: ~40% of queries):
- Multi-step tasks: "implement a function", "debug the error"
- Questions with 15-30 words
- Tasks mentioning specific files or functions

**Complex** (target: ~20% of queries):
- Architecture decisions: "design a system", "plan architecture"
- Refactoring: "refactor the authentication module"
- Long queries (> 30 words)
- Keywords: "architect", "migrate", "redesign", "comprehensive", "strategy"

### Adaptive Pre-Stream Processing

#### Conditional Memory Recall

```python
async def _maybe_recall_memory(
    self,
    user_input: str,
    complexity: ComplexityLevel,
) -> list[MemoryItem]:
    """Recall memory only for medium/complex queries."""
    if complexity not in ("medium", "complex"):
        return []

    if not self._memory:
        return []

    return await self._memory.recall(user_input, limit=5)
```

**Rationale**: Trivial and simple queries don't benefit from historical context. Skipping saves 1-2 seconds.

#### Conditional Context Projection

```python
async def _maybe_project_context(
    self,
    user_input: str,
    complexity: ComplexityLevel,
    token_budget: int = 4000,
) -> ContextProjection | None:
    """Project context only for medium/complex queries."""
    if complexity not in ("medium", "complex"):
        return None

    if not self._context:
        return None

    return await self._context.project(user_input, token_budget=token_budget)
```

**Rationale**: Context projection requires ranking and token allocation. Skipping saves 0.5-1 second for queries that don't need it.

#### Adaptive Plan Creation

```python
async def _create_plan_adaptive(
    self,
    goal: str,
    complexity: ComplexityLevel,
    context: PlanContext,
) -> Plan:
    """Create plan with adaptive strategy based on complexity."""
    # Trivial: Single-step template
    if complexity == "trivial":
        return Plan(
            goal=goal,
            steps=[PlanStep(id="step_1", description=goal, execution_hint="auto")],
        )

    # Simple/Medium: Try template, fall back to LLM
    if complexity in ("simple", "medium"):
        template_plan = self._match_plan_template(goal)
        if template_plan:
            return template_plan

    # Complex or template miss: Use LLM
    return await self._planner.create_plan(goal, context)
```

### Parallel Execution

When both memory and context are needed, execute in parallel:

```python
async def _pre_stream_parallel(
    self,
    user_input: str,
    complexity: ComplexityLevel,
) -> tuple[list[MemoryItem], ContextProjection | None]:
    """Run memory and context operations in parallel."""
    if complexity not in ("medium", "complex"):
        return ([], None)

    tasks = []

    # Memory recall task
    if self._memory:
        tasks.append(self._memory.recall(user_input, limit=5))
    else:
        tasks.append(asyncio.sleep(0, result=[]))

    # Context projection task
    if self._context:
        tasks.append(self._context.project(user_input, token_budget=4000))
    else:
        tasks.append(asyncio.sleep(0, result=None))

    # Execute in parallel
    memory_items, context_projection = await asyncio.gather(*tasks)

    return memory_items, context_projection
```

**Savings**: Overlaps 1-2s of I/O-bound work, saving 500ms-1.5s.

### Caching Strategies

#### Embedding Cache

```python
from functools import lru_cache
from hashlib import sha256

class MemoryProtocol:
    @lru_cache(maxsize=100)
    def _get_cached_embedding(self, text_hash: str) -> list[float] | None:
        """Retrieve cached embedding by text hash."""
        return None  # Override in implementation

    async def recall(self, query: str, limit: int = 5) -> list[MemoryItem]:
        """Recall with embedding cache."""
        query_hash = sha256(query.encode()).hexdigest()

        # Try cache
        cached_embedding = self._get_cached_embedding(query_hash)
        if cached_embedding is not None:
            return await self._search_with_embedding(cached_embedding, limit)

        # Generate and cache
        embedding = await self._generate_embedding(query)
        # Cache populated on next call
        return await self._search_with_embedding(embedding, limit)
```

**Savings**: 200-800ms for cache hits (skip embedding generation).

#### Plan Template Cache

```python
class PlannerProtocol:
    _PLAN_TEMPLATES = {
        "question": Plan(
            goal="",
            steps=[PlanStep(id="step_1", description="", execution_hint="auto")],
        ),
        "search": Plan(
            goal="",
            steps=[
                PlanStep(id="step_1", description="Search for information", execution_hint="tool"),
                PlanStep(id="step_2", description="Summarize findings", execution_hint="auto"),
            ],
        ),
        "analysis": Plan(
            goal="",
            steps=[
                PlanStep(id="step_1", description="Analyze the content", execution_hint="auto"),
                PlanStep(id="step_2", description="Provide insights", execution_hint="auto"),
            ],
        ),
    }

    def _match_plan_template(self, goal: str) -> Plan | None:
        """Match goal to predefined template."""
        import re
        goal_lower = goal.lower()

        if re.match(r'^(who|what|where|when|why|how)\s+', goal_lower):
            return self._PLAN_TEMPLATES["question"].model_copy(deep=True)

        if re.match(r'^(search|find|look up)\s+', goal_lower):
            return self._PLAN_TEMPLATES["search"].model_copy(deep=True)

        if re.match(r'^(analyze|analyse|review|examine)\s+', goal_lower):
            return self._PLAN_TEMPLATES["analysis"].model_copy(deep=True)

        return None
```

**Savings**: 1.5-2.5s for template-matched queries (no LLM call).

## Implementation Considerations

### Backward Compatibility

- Feature flags default to `false` (current behavior preserved)
- Gradual rollout: enable optimizations incrementally
- Existing API surface unchanged
- No changes to protocol interfaces

### Testing Strategy

1. **Unit tests**:
   - Query classifier accuracy
   - Template matching correctness
   - Cache behavior

2. **Integration tests**:
   - End-to-end latency by complexity
   - Parallel execution correctness
   - Feature flag combinations

3. **Performance benchmarks**:
   - Baseline measurements
   - Optimization impact
   - Regression detection

### Failure Modes

| Failure | Mitigation |
|---------|-----------|
| Classifier error | Default to "simple" (safe middle ground) |
| Cache miss | Fall back to full computation |
| Template mismatch | Fall back to LLM planning |
| Parallel task failure | Return partial results, log error |

### Monitoring and Observability

1. **Logging**:
   - Phase timing at INFO level
   - Classification decisions at DEBUG level
   - Cache hit/miss at DEBUG level

2. **Metrics**:
   - Performance counters per phase
   - Complexity distribution histogram
   - Cache hit rate gauges

3. **Alerting**:
   - Latency SLA violations
   - Cache hit rate drops
   - Classification accuracy degradation

## Performance Targets

### Latency SLAs

| Complexity | P50 | P90 | P99 |
|------------|-----|-----|-----|
| Trivial | 300ms | 500ms | 800ms |
| Simple | 700ms | 1s | 1.5s |
| Medium | 1.5s | 2s | 3s |
| Complex | 2.5s | 3s | 4s |

### Quality Targets

- No regression in response quality for complex queries
- Template plan acceptance rate > 95% for matched queries
- Classification accuracy > 90% across all complexity levels

## Future Extensions

### Potential Enhancements

1. **ML-based classification**: Replace heuristics with lightweight classifier
2. **Predictive caching**: Pre-load resources for predicted next queries
3. **Streaming context**: Project context incrementally during stream phase
4. **Dynamic thresholds**: Adjust complexity thresholds based on usage patterns

### Research Directions

1. **User intent modeling**: Better understand query types
2. **Cost-quality tradeoffs**: Measure quality impact of optimizations
3. **Adaptive strategies**: Learn optimal processing strategies from usage

## Security Considerations

- Classification must not leak sensitive information
- Caching must respect data isolation between threads
- Template matching must not expose internal patterns
- Performance metrics must not log sensitive query content
