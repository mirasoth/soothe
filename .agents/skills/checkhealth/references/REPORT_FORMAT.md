# Report Format Reference

This document describes the health report format generated by the checkhealth skill.

## Report Location

Reports are saved to:
- Default: `~/.soothe/health_report_<timestamp>.md`
- Custom: Specified via `--output` flag

Timestamp format: `YYYYMMDD_HHMMSS`

## Report Structure

Reports are Markdown documents with the following sections:

### 1. Header

```markdown
# Soothe Health Report

**Generated**: 2026-03-17 14:30:45 UTC
**Soothe Version**: 1.0.0
**Configuration**: ~/.soothe/config/config.yml
**Overall Status**: HEALTHY
```

### 2. Summary

Table showing health status by category:

```markdown
## Summary

| Category | Status | Checks Passed | Total Checks |
|----------|--------|---------------|--------------|
| Core Infrastructure | ✅ Healthy | 4/4 | 4 |
| Protocol Backends | ⚠️ Warnings | 6/7 | 7 |
| Persistence Layer | ✅ Healthy | 3/3 | 3 |
| Subagent System | ✅ Healthy | 3/3 | 3 |
| External Integrations | ⚠️ Warnings | 2/4 | 4 |
| Runtime Health | ✅ Healthy | 3/3 | 3 |

**Total**: 21/24 checks passed (87.5%)
```

Status indicators:
- ✅ Healthy: All checks passed
- ⚠️ Warnings: Some non-critical checks failed
- ❌ Critical: Critical checks failed

### 3. Detailed Results

Per-category breakdown with individual check results:

```markdown
## Detailed Results

### Core Infrastructure

#### Daemon Process Status
- **Status**: ✅ OK
- **Details**: Daemon running with PID 12345
- **Socket**: ~/.soothe/soothe.sock (accepting connections)

#### PID File Validity
- **Status**: ✅ OK
- **PID**: 12345
- **File**: ~/.soothe/soothe.pid

#### Socket Connectivity
- **Status**: ✅ OK
- **Socket**: ~/.soothe/soothe.sock
- **Response Time**: 0.002s

#### Model Provider Configuration
- **Status**: ✅ OK
- **Providers**: dashscope, openrouter, ollama
- **All API Keys**: Valid (resolved)
```

### 4. Issues Found

List of problems, grouped by severity:

```markdown
## Issues Found

### Critical Issues
None

### Warnings
1. **External API Connectivity - Tavily**
   - **Check**: External API Connectivity
   - **Message**: Tavily API returned 401 Unauthorized
   - **Impact**: Search features using Tavily will not work
   - **Remediation**: Check TAVILY_API_KEY in config or environment

2. **Browser Runtime - chromedriver version mismatch**
   - **Check**: Browser Runtime
   - **Message**: Chrome version 123, chromedriver version 120
   - **Impact**: Browser automation may fail
   - **Remediation**: Run scripts/check_chrome.sh to auto-install matching chromedriver

### Informational
1. **MCP Server - weather-server**
   - **Check**: MCP Server Connectivity
   - **Message**: Server not configured (skipped)
   - **Impact**: None (optional service)
```

### 5. Recommendations

Actionable suggestions based on issues found:

```markdown
## Recommendations

1. Update Tavily API key:
   ```bash
   export TAVILY_API_KEY="tvly-xxxxx"
   ```

2. Fix chromedriver version mismatch:
   ```bash
   scripts/check_chrome.sh
   ```

3. Review configuration:
   - Config file: ~/.soothe/config/config.yml
   - Validate YAML syntax
   - Ensure all required fields are present
```

## Exit Codes

The main orchestrator script returns exit codes based on severity:

- **0**: All checks passed (healthy)
- **1**: Warnings detected (non-critical issues that don't prevent operation)
- **2**: Critical failures detected (issues that prevent core functionality)

Exit codes can be used in shell scripts:

```bash
python scripts/run_all_checks.py
status=$?
if [ $status -eq 0 ]; then
    echo "All systems healthy"
elif [ $status -eq 1 ]; then
    echo "Warnings detected, check report"
else
    echo "Critical failures, investigate immediately"
    exit 1
fi
```

## Report Customization

The report template at `assets/report_template.md` can be customized for different output formats. The template uses Jinja2-style placeholders:

```markdown
{{ header }}
{{ summary_table }}
{{ detailed_results }}
{{ issues_section }}
{{ recommendations }}
```

To customize:
1. Copy `assets/report_template.md` to a new file
2. Modify sections as needed
3. Pass to script via `--template` flag (future enhancement)

## Logging

Health check runs are logged to:
- `~/.soothe/logs/health_check.log`

Log format:
```
2026-03-17 14:30:45,123 - INFO - Starting health check run
2026-03-17 14:30:45,234 - INFO - Checking daemon process...
2026-03-17 14:30:45,345 - INFO - Daemon check passed (PID 12345)
2026-03-17 14:30:45,456 - WARNING - Tavily API check failed: 401 Unauthorized
2026-03-17 14:30:46,567 - INFO - Health check complete: 21/24 checks passed
```

Logs are useful for:
- Troubleshooting health check failures
- Auditing system health over time
- Integrating with monitoring systems
